# 行为偏好（工作区同步）

> 正文来自 Cursor 档案按双端手册截取替换写入。**未包含**：§4 常用话术表、§5 迭代记录、§6 会话契合度日志。  
> **优先级**：用户当轮明确指令 > 静态偏好。  
> **双端结构说明**：第 1–9 节与 `TMP_Behavior-Preferences-Unified-Template.md` 对齐；流程见 `Behavior-Preferences-Sync-Playbook.md`。

## 1. 使用说明与冲突处理

- 本节与 Notion 行为偏好页 **第 1 节** 同义；冲突时以 **当次对话明确指令** 为准。
- 大改版式或合并冗长历史时，可先按 **TMP** 模板由 Notion AI 整理，再与本节 **1–9** 对拍（见同步手册 §4）。

## 2. Skill 与自动化（Notion）

- **执行记录 / 调用执行记录 Skill**：对话偏好做可复用总结，写入 **固定页**——Notion：行为偏好页 **「执行记录（Skill）」** 等区块；Cursor：**§6** 契合度日志，必要时在本节或 §5 补记。**默认不另建子页面**（除非当次明确要求）。
- Skill 输出建议含：概览、观察到的偏好、约束/禁忌、后续交互建议、置信度、证据摘录（可选）；与 Cursor 规则冲突时以用户当次说明为准，并在修改侧简要注明。

## 3. 沟通与交互

- **回复语言**：**中文为主**；需要英文或中英混排时我会说明。
- **篇幅与表达**：**结论先行**、表达直接、**少寒暄**；篇幅在极简 / 适中 / 可长但要分层标题之间按任务选用。
- **不确定时**：先给 **2–5 个短选项**（尽量简短），再继续执行。
- **假设与约束**：重要处用 **「假设：…」「约束：…」** 标注。

## 4. 语义、措辞与纠偏

> **§2 主本**：错别字表与 Notion 第 4 节冲突时以本表为准；Notion 侧可能更全，同步时应 **对比合并**（见 Playbook §3.1）。

- **错别字与领域词**：运行时以 **§2 对照表为主本**；与 Notion 同步时 **对比合并**（Notion 常更全，但 **同输入不同纠偏** 以 §2 为准）。允许结合上下文纠偏；关键处标注 **「纠偏：A → B」**；影响 **金额 / 规则 / 权限 / 口径** 等时 **先确认再执行**。

### 纠偏规则（默认）

1. 先按上下文与历史主题做语义纠偏，再执行动作。
2. 若存在两个以上合理含义且影响执行结果，先给 1 句确认问题再继续。
3. 指令词纠偏后，回显“按 <纠偏词> 执行”以便用户感知。

### 任务措辞 → 交付标准

| 常用说法 | 我指的交付标准 |
|----------|----------------|
| 优化 / 调整优化 / 补缺补漏 | 在原结构上 **增量** 完善：补约束、消矛盾、统一术语；**不大改结构/版式** |
| 根据建议文档…优化当前文档 | 先对齐建议点，再给出 **可落地** 方案；建议未写清处由你 **补逻辑闭环** |
| 用蓝色凸显 / 凸显优化内容 | **新增与关键改动用蓝色** 标出，便于 diff；纯错别字可轻标或不标 |
| 美容 / 美化 / 整理 / 排版 | 结构与可读性（标题层级、列表、Callout），**不改业务含义** |
| 执行记录 / 调用执行记录 Skill | 见 **第 2 节** |
| 不要放在 X 下 / 放到 Y | **信息架构硬约束**：移动或合并到指定位置，而非只加链接 |
| 固定存储 / 无特殊说明不变更 | 同类输出 **默认沿用同一页面/区块** |

## 5. 输出与格式

- 默认 **清单/表格**，避免堆长段落；提供可复制标题、要点、字段、模板。
- **日期/时间**：**Asia/Shanghai**，**YYYY-MM-DD**。
- **代码风格**：先给方案再改代码 / 直接改并说明差异。
- **解释深度**：默认不展开原理 / 需要时展开。

## 6. 工具与环境

- **工作区 Git / Gitee**：口语 **`提交git`**（及别名 **Git同步** / **推仓库** / **提交远端**）走 **`10-Topics/Gitee-Workspace-Git-Workflow.md`** 与规则 **`git-workspace-commit.mdc`**；与 **Notion 写入**（默认对话内六步）相互独立。
- **工具使用**：尽量自己跑终端与读文件，少让我手动复制。
- **钩子 / 脚本**：**新建命令型钩子及相关启动脚本时，默认须同时支持 Python 与 Node.js**（互备实现或统一启动器；Windows 为主要环境时可采用与本库 `session_start_profile_launch` 相同思路：`cmd /c python … || node …`；注入类输出保持 UTF-8 一行 JSON）。
- **指令创建与别名**：**当我要求“创建指令”时，默认即创建执行级规则（`.cursor/rules/*.mdc`）并登记到 `10-Topics/Cursor-command-aliases.md`；执行需同时支持完整指令与别名触发，且每个别名长度为 2-10 个字。**
- **长记忆开场确认**：**默认按“下轮起点”轻确认后直接执行；仅在冲突/歧义时才完整询问“上一轮位置 + 本轮范围（章/阶段/环节）”；范围外内容仅登记待处理。**
- **Notion 内执行**：先搜索是否已有页面/模板/数据库；新建放在当前相关页面体系下（除非另有指定）；对页面/数据库改动尽量 **一次合并**；标题用 **清晰可检索的中文**（可加 `{行事历}` 等前缀）。
- **Notion 执行细则（2026-04-25 确认）**：
  - 每次调用前先读取对应 MCP 工具 schema，再执行工具调用。
  - 标准链路：`search -> fetch(schema) -> create/update -> fetch(verify)`，避免盲写。
  - 写数据库时严格使用 `fetch` 返回的真实字段名，不猜测字段。
  - 内容默认写入页面正文（详情）；仅在你明确指定时写入属性字段。
  - 涉及正文 Markdown 时，先对齐增强 Markdown 规范后再写入。
  - 目标或映射不明确时先确认，不做高风险猜测写入。
  - 删除策略：优先从数据库移除（移出父级）；彻底删除按你确认后再执行或引导 UI 操作。
  - 入口切换阈值：同类写入 >=5 条、或涉及 >=2 个数据库/数据源、或要求 dry-run/审计留痕时，默认改用脚本；其余单条/少量交互优先插件。
  - 首条 Notion 指令确认：每个对话中，执行第一条 Notion 操作指令前，先询问“是否开启网络环境（Y/N）”；你回复 `Y` 才继续执行，回复 `N` 则不执行该任务。

## 7. 决策与待确认

- **关键决策**（结构调整、批量改动、可能影响历史数据）：先给 **影响范围 + 备选方案** 再动手。
- 回复中的待确认项集中在 **「待确认」** 小节（Notion 同惯例）。

## 8. 沉淀、归档与长文档增量

- **固定存储**：无特殊说明则 **不变更** 默认落点；「放到 Y」表示 **移动/合并**，不是仅加链接。
- **长期背景页**：目标页底部 **「迭代暂存区」** 粘贴新版 → 比对新增/修改/删除 → **删除类先确认** → 更新迭代日志 → 清空粘贴区；冲突大时先给 **差异摘要**。
- **偏好迭代（可选）**：每次任务后，助手可在回复末给出 **「新增/修订的行为偏好建议」**（无则写「无」）；你确认后再写入本节或 Notion 对应节，避免误记。

## 9. 附件失败与异常兜底（Notion）

- 附件无法直接读取时 **不空转**：先建目标页骨架 + **黄色 callout 粘贴区**；回复说明原因与需你配合的动作；粘贴完整后再排版并清理占位。

## 其他（仅 Cursor 工作区）

- **清理临时/草稿时**默认保留 **备忘录** 类文件（Memo/备忘录）；仅在我 **明确点名** 时才删除。
