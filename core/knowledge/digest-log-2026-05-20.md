## 逐协议消化说明

### CP-098 — 系统基模图鉴

- **核心主张**：10 个系统基模是重复出现的结构模式——识别基模 = 从"治标"转向"治本"
- **主分类**：structural（框架性）——提供分析问题结构的多维视角
- **自我领悟**：系统基模用增强回路+调节回路的组合描述结构性陷阱，每个基模都有明确的"杠杆点"——不是在症状上使劲
- **评断验证**：PASS（三源交叉验证：Senge 原书 + MIT CTL 讲义 + Systems Thinker 图鉴）
- **confidence**：high（三源交叉验证一致）

### CP-099 — 因果回路图与存量流量图

- **核心主张**：CLD 定性建模（快速沟通反馈结构）+ SFD 定量建模（精确仿真系统行为），两者互补——先 CLD 建共识，再 SFD 验假设
- **主分类**：method（程序性）——提供建模的步骤规范
- **自我领悟**：CLD 的核心陷阱是变量必须可增减、延迟必须标注、区分实际vs感知——这些不是"格式要求"，是"模型是否会得出错误结论"的关键
- **评断验证**：PASS（Cascade Institute 手册 + Systems Thinker 多篇文章 + 存量流量转换步骤三源一致）
- **confidence**：high（三源交叉验证一致）

### CP-100 — 十二杠杆点

- **核心主张**：12 级干预点按效力递增——参数最低，范式最高。反直觉：95% 注意力在最无效的杠杆点上
- **主分类**：structural（框架性）——提供干预效力的分层视角
- **自我领悟**：Meadows 的"系统阻力随杠杆升高而升高"——越有效的地方系统越会抵抗。这对理解"为什么 Phoenix 系统改进总是堵在低杠杆层"非常重要
- **评断验证**：PASS（Meadows 1997 原文 + Wikipedia 百科 + Dartmouth 教学材料三源一致）
- **confidence**：high（三源交叉验证一致）

### CP-101 — Cynefin 五域决策框架

- **核心主张**：先判域（清晰/繁杂/复杂/混沌/失序）→ 再匹配策略。域错了，再好的方法也反效果
- **主分类**：method（程序性）——提供域判别的决策步骤
- **自我领悟**：CP-101 与 P008 互补——P008 问"能不能自动做"，Cynefin 问"应该怎么想"
- **评断验证**：PASS（Snowden 原始框架 + Cynefin.io 官方文档 + SFU 学术材料三源一致）
- **confidence**：high（三源交叉验证一致）

### CP-102 — 决策树与贝叶斯推理

- **核心主张**：决策树用树形结构量化不确定下的选择（期望值回推），贝叶斯用新证据持续更新信念（P(H|E)更新）
- **主分类**：method（程序性）——提供结构化的决策分析步骤
- **自我领悟**：决策树是"结构"（列出所有路径），贝叶斯是"燃料"（校准概率）。两者互补——树没有概率是空的，概率没有树不知道用于什么决策
- **评断验证**：PASS（Raiffa/Schlaifer 1961 经典 + 现代应用总结一致）
- **confidence**：medium（双源验证一致，第三源未找到独立的树-贝叶斯联动教学材料）

### CP-103 — RACI 角色责任矩阵

- **核心主张**：R(执行)/A(审批=唯一)/C(咨询)/I(知情) 四角色——"每个任务有且只有一个 A"
- **主分类**：method（程序性）——提供团队角色分配的步骤规范
- **自我领悟**：RACI 最核心的规则是 A 唯一性——多人共担 = 无人真正负责。多 A 是形式正确但实质失效的最常见陷阱
- **评断验证**：PASS（Adobe + Smartsheet + Wrike 三源一致）
- **confidence**：high（三源交叉验证一致）

## ① 审视现有系统 — 改进启示

### 启示 1：规则膨胀的"转移负担"基模

**对应协议**：CP-098（系统基模·转移负担）

**现象**：
```
问题：LLM 遗漏步骤/跳过确认（系统行为不稳定）
症状解：加更多 .mdc 规则（现 40+ 条）
副作用：规则越多 → 上下文窗口越挤 → 单条规则遵守率下降 → 需要更多规则
根本解：Guard 代码化——规则从"自然语言建议"变成"代码强制执行"
```

**建议**：Guard 到位前的过渡期，每次新增规则时须声明"这条规则的约束在 Guard 落地后是否可以代码化？"——若是 → 加 `guard_replaceable: true` 标注，防止临时措施变永久债务。

### 启示 2：系统目标缺失（杠杆点第 3 级）

**对应协议**：CP-100（十二杠杆点·目标）

**现象**：Phoenix 系统的目标隐含在 40+ 条规则里（"帮用户高效完成任务""防止 Agent 越权""确保数据一致性"……），没有单一清晰的目标声明。

**建议**：在 Guard 集成（P035）时，设计一条"系统目标声明"——在 Guard 启动时作为顶层指令注入给 LLM#1。格式参考 Meadows："这个系统的目标是 [单一陈述]。所有下游决策（P008 评估等级、KB 预载入范围、工具选择优先级）都以此为校准锚点。"

### 启示 3：系统角色缺失 RACI 定义

**对应协议**：CP-103（RACI 矩阵）

**现象**：Guard 引擎、LLM、Cursor Agent 三者的角色边界不清晰。当一个任务涉及 KB 预载入 + P008 评估 + Skill 调用时，谁对"最终执行结果"负 A（最终问责）？

**建议**：P035 双轨过渡方案中，补一个 Guard-Cursor 双轨 RACI 矩阵：

| 任务 | R（执行者） | A（审批人） | C（咨询者） | I（知情者） |
|:---|:---|:---|:---|:---|
| 任务意图感知 | Guard | — | LLM#1 | Cursor Agent |
| P008 风险评估 | Guard | 用户（≥L2 时） | .mdc 规则 | Cursor Agent |
| KB 协议预载入 | Guard | — | Knowledge-Brain | Cursor Agent |
| 任务执行 | Cursor Agent | Guard（安全闸门兜底） | Guard（注入上下文） | 用户（告知执行结果） |

### 启示 4：Cynefin + P008 互补判域

**对应协议**：CP-101（Cynefin） + CP-098（系统基模）

**现象**：P008 评估一个任务为 L0（全自动），但如果它在 Cynefin 的"复杂域"（因果不可预测），直接自动执行可能方向就错了。

**建议**：在 task-init-protocol.mdc 的 Step 1（理解任务）中，加一句轻量 Cynefin 判域提示——不是完整五域分析，只需判断"因果可预测（清晰/繁杂）还是不可预测（复杂/混沌）"。复杂域任务即使 P008 评估为 L0，也建议"先做小范围探测再铺开"。
