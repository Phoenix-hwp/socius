# 2026-05-12 D盘清空事故备忘

> **严重程度**：灾难级 — 工作区及 D 盘大部分文件资料被删除。
> **根本原因**：Skill 试运行目录清理时，强制杀掉占用进程导致 shim 崩溃、模型断连，新模型在上下文断裂状态下执行递归删除，CWD 异常导致级联扩大到整盘。
> **状态**：工作区已通过 git clone 完整恢复；非 Git 追踪文件永久丢失；复盘后完成 5 项规则加固。

## 2. 根因分析（三层失效）

### 第一层：缺少试运行前落锚

试运行前未强制提交 Git，无快速恢复基线。若有基线，事故后可 `git reset --hard <基线>` 一键恢复。

### 第二层：清理方式错误

规则规定用 `git worktree remove`（安全、精确），实际走了文件系统递归删除命令（`rmdir /s /q` 或等效），且**没有命令预览和 CWD 确认**。

### 第三层：进程杀法错误 + 上下文断裂

- 杀进程时用了通配进程名（`taskkill /f /im node.exe`），误杀了 shim 和 Cursor 依赖
- 模型断连后新模型无历史上下文，不知道删除命令的精确目标和约束
- 新模型在 CWD 非预期的情况下继续执行，「递归删除试运行目录」变成了「递归删除当前目录及所有子目录」

## 5. 教训

1. **清理操作不是「简单操作」**：看似无害的文件删除，在高风险上下文（进程占用、Git worktree、模型断连可能）中可演变为灾难
2. **隔离墙不能只靠文件系统**：worktree 提供目录隔离但不提供进程隔离；落锚（Git 基线）是最后一道防线
3. **杀进程必须精确到 PID**：通配进程名杀进程在网络隧道（ngrok + shim）环境中等于自杀
4. **模型断连后新 Agent 无历史是高风险状态**：此时应暂停所有破坏性操作，先恢复上下文
5. **对话备份与事故记录比代码更重要**：代码有 Git 可恢复；对话上下文和事故日志丢了就是永久丢失
