# 系统基模图鉴

## 核心概念

**系统基模**（Systems Archetypes）是 Peter Senge 在《第五项修炼》中系统化的一组重复出现的系统结构模式。它们是"看清整体的学科"——识别事物之间的关系而非孤立的事物本身。

每个基模由两类回路构成：
- **增强回路（R）**：自我强化，推动增长或崩溃
- **调节回路（B）**：自我修正，追求目标或稳定

基模的用途是双重的：**诊断**（识别已有的问题模式）和**预警**（预见未来可能的非预期后果）。

---

## 十大核心基模

### 1. 增长极限（Limits to Growth）

```
    增强回路（R）驱动增长 → 但增长本身触发 → 调节回路（B）开始刹车
    结果：S 型曲线（先增长后饱和）或 超越—崩溃
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 一开始增长很快，后来渐渐慢下来，最后停滞或下滑。越努力推动增长，越加速撞墙 |
| **杠杆点** | **不要更用力推增强回路**（那会加速撞墙）。找到并消除限制因素——通常不在增长引擎所在处，而在系统的另一部分 |
| **示例** | 初创公司靠加班冲业绩 → 加班导致倦怠/质量下降 → 客户流失 → 更用力加班（恶性循环） |

**应对原则**：增长的杠杆不在增长引擎里，在限制因素上。问"什么东西在刹车？"而非"怎么加更多油门？"

---

### 2. 转移负担（Shifting the Burden）

```
    问题症状 → 症状解（快但治标）→ 症状缓解（但根本原因未消除）
            → 根本解（慢但治本）→ 但症状解的"成功"削弱了对根本解的压力
    症状解产生副作用 → 根本解的能力被侵蚀 → 越来越依赖症状解
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 同一个问题反复出现，每次都用"快速修复"解决，但问题越来越严重，组织的根本能力在衰退 |
| **杠杆点** | **同时使用症状解和根本解**——症状解争取时间，根本解建立长期能力。关键是承认对根本解的投资需求，防止症状解的"成功"让人忘记根本解 |
| **示例** | 依赖外包解决技术债务（症状解）→ 内部工程能力萎缩 → 未来需要更多外包 → 恶性循环。正确做法：外包 + 同时建立内部能力 |

**核心警示**：上瘾结构——对症状解的依赖越深，实施根本解的能力越弱。

---

### 3. 饮鸩止渴（Fixes That Fail）

```
    问题 → 短期修复（有效！）→ 但产生非预期的副作用 → 长期使问题更严重
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 措施在短期内有效，但问题在长期内变得更糟，且往往以新的、更棘手的形式出现 |
| **杠杆点** | 在采取行动前，先问"这个方案的长期副作用是什么？"优先选择没有副作用的方案。如果副作用不可避免，提前设计应对措施 |
| **示例** | 砍研发预算 → 短期利润提升 → 产品老化 → 市场份额下滑 → 更狠地砍预算 |

**与转移负担的区分**：饮鸩止渴的"修复"直接产生副作用使问题恶化；转移负担是症状解侵蚀了根本解的能力。

---

### 4. 公地悲剧（Tragedy of the Commons）

```
    多个行动者各自理性地增加消耗 → 共享资源逐渐枯竭 → 所有人一起遭殃
    A 的收益 = 个人获得 → A 的成本 = 所有人分摊 → 理性个体 = 过度消耗
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 共享资源（代码库质量、团队注意力、公共预算）持续退化，每个个体都在做"理性"但总体不可持续的选择 |
| **杠杆点** | ① 教育个体理解集体后果；② 建立共享资源的反馈机制（让消耗者看到总量）；③ 设立使用上限或产权界定 |
| **示例** | 多个团队共用同一个 QA 资源池，各团队都想"多用一点"→ QA 过载质量下降 → 所有人受损失 |

---

### 5. 强者愈强（Success to the Successful）

```
    两个竞争实体 → A 获得比 B 更多的初始资源 → A 表现得更好 → A 获得更多资源 → B 获得更少 → B 表现更差 → 差距持续扩大
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 赢家持续赢，输家持续输，不是因为能力差异，而是因为"初始优势 → 资源倾斜 → 能力差异固化"的循环 |
| **杠杆点** | ① 在竞争中引入平衡机制（如累进税反对财富集中）；② 多元化评估标准，不只用一个指标决定资源分配；③ 给弱势实体发展能力的独立路径 |
| **示例** | 明星产品拿到所有营销预算 → 其他产品没有曝光 → 其他产品表现差 → 结论："我们只有一个好产品"（自证预言） |

---

### 6. 恶性竞争 / 军备竞赛（Escalation）

```
    A 认为 B 超过了自己 → A 加大投入 → B 感到威胁 → B 加大投入 → A 再次超越 → 螺旋升级
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 双方不断加大投入但相对位置从未改变，消耗却持续增加。双方都认为对方是问题来源 |
| **杠杆点** | **单方面打破循环**——一方主动降级。或双方协商建立上限（如军备控制条约）。关键是识别这个循环不是"竞争"，而是"共同毁灭" |
| **示例** | 两家公司对标同一功能 → A 加功能 → B 跟功能 → A 再加 → 功能膨胀但用户不在意 → 两家都浪费了研发资源 |

---

### 7. 目标侵蚀（Eroding Goals）

```
    理想目标与现实表现有差距 → 降低目标（而非改进表现）→ 差距消失 → 表现继续下滑 → 再降低目标
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 标准/目标不断地下调，"现实一点"的说法频繁出现。长期趋势是滑坡式的 |
| **杠杆点** | **锁定目标**——将目标与外部基准（行业标准/客户期望）绑定，而非与自己的历史表现比较。设立"不允许低于"的底线 |
| **示例** | "交付质量 90% → 发现做不到 → 把目标改成 80% → 做不到 → 改成 70% → ..."——组织能力的真正衰退 |

---

### 8. 增长与投资不足（Growth & Underinvestment）

```
    需求增长 → 需要扩大产能 → 但投资不足（因"需求不确定"或"短期利润优先"）→ 服务质量下降 → 需求增长放缓 → 进一步"证明"不需要投资 → 恶性循环
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 增长阶段的服务/质量持续恶化，但"数据"显示不需要投资（因为需求在恶化后确实降了）。这是增长极限的高级变体——限制因素是可被投资消除的 |
| **杠杆点** | 在需求增长初期就投资建设能力——如果等标准恶化到"数据支持投资"时已经太晚。关键：基于领先指标而非滞后指标做投资决策 |
| **示例** | 用户快速增长 → 服务器来不及扩容 → 体验变差 → 用户增长放缓 → "你看，增长放缓了，不需要扩容" |

---

### 9. 意外对手（Accidental Adversaries）

```
    A 和 B 本是互补合作 → A 的行动（无意中）伤害了 B → B 采取防御措施 → 防御措施（无意中）伤害了 A → A 更强地反击 → 合作变对抗
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 曾经的合作伙伴渐渐变得敌对，双方都抱怨对方"变了"。但双方的本意都不是对抗——是各自解决自身问题的行动无意间伤害了对方 |
| **杠杆点** | 重建双方的"合作成功"回路——让双方重新看到合作带来的共同收益大于各自单飞的收益。透明的沟通和共同目标的重新对齐 |
| **示例** | 产品团队和工程团队——产品为实现业务目标不断加需求 → 工程质量恶化 → 工程团队设卡限制 → 产品觉得工程"不配合" → 绕过工程自己推 |

---

### 10. 吸引力原则（Attractiveness Principle）

```
    面对多个互相冲突的解决方案 → 选择看起来最有吸引力的（但通常最治标的）→ 短期满意 → 长期问题未解决 → 下一次仍选有吸引力的方案
```

| 要素 | 描述 |
|:---|:---|
| **典型症状** | 组织在各种方案间跳来跳去，每次选"最有吸引力的"，但底层问题从未解决。有吸引力的方案通常：快见效、省力、低风险 |
| **杠杆点** | 识别"吸引力"背后的短期主义动机。在决策时明确区分"我选择这个是因为它真正有效"和"我选择这个是因为它让我此刻感觉好" |
| **示例** | 技术选型总是选最热门的框架 → 几个月后发现不适合 → 换下一个热门框架 → 底层架构问题从未解决 |

---

## 基模识别三步法

1. **画回路**：画出核心变量和它们之间的因果箭头（+/-），识别增强回路和调节回路
2. **找模式**：观察回路组合后产生的动态行为（S型增长？震荡？螺旋升级？）
3. **对基模**：将观察到的动态行为与以上 10 个基模比对，找到最匹配的结构模式

**关键提醒**：基模是"结构模式"而非"分类标签"。同一系统中的不同问题可能对应不同基模；一个系统的问题可能同时涉及多个基模。
