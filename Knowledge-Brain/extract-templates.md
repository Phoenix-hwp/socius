# 知识提炼模板

## 提炼总则

1. **去叙留骨**：从叙述性知识中抽掉例子和背景，留下可复用的结构骨架。
2. **保持单一性**：一个提炼单元只处理一个概念/一个方法/一个经验。避免一个协议混杂多种知识类型。
3. **概念锚必填**：每条提炼结果必须标注 `concept_anchor`。
4. **提炼不替代原文**：提炼产物是「骨架」，溯源链接指向原文供按需回顾。

## 一、概念性知识（是什么）

### 提炼骨架

```
定义 → 要素/层次 → 边界 → 关联概念
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 定义 | 一句话精确定义 + 中文/英文术语名 | 是 |
| 核心要素 | 组成该概念的要素（列表，每个要素 ≤1 行解释） | 是 |
| 层次/分类 | 该概念内部的层次结构或分类体系（若存在） | 否 |
| 边界 | 什么不是它；与邻近概念的区分红线 | 是（至少 1 条） |
| 关联概念 | 相关概念及关系类型（上位/同级/下游/前置） | 否 |

### 完整示例

**原始知识**：BPMN 活动是业务流程中执行的工作单元，包含任务（原子）、子流程（复合可展开）和编排任务（跨参与者消息交互）三层次，并有服务/用户/手动/业务规则/脚本/发送/接收等多种任务类型标识。

**提炼产出**：

```markdown
# CP-xxx：BPMN 活动

## 定义
BPMN 活动（Activity）是业务流程中执行的工作单元，是 BPMN 四类流对象之一。

## 核心要素
- 是「流程中做什么」的载体
- 以圆角矩形表示
- 可叠加循环、多实例、补偿等行为标记

## 层次
| 层次 | 特征 | 符号 |
|:---|:---|:---|
| 任务（Task） | 原子活动，无法进一步细分 | 圆角矩形 |
| 子流程（Sub-Process） | 可展开为内部流程 | 圆角矩形 + 展开标记 |
| 编排任务（Choreography Task） | 涉及两个参与者的消息交换 | 上下带区显示参与者 |

## 边界
- 不是事件（事件是「发生了什么」，活动是「做什么」）
- 不是网关（网关是「走哪条路」，活动是「路上做什么」）
- 不是数据对象（数据是「什么信息在流转」，活动是「谁在流转中处理信息」）

## 关联概念
- 上位：BPMN.flow_element（流元素）
- 同级：BPMN.gateway（网关）、BPMN.event（事件）
- 下游：BPMN.activity sub-types（任务/子流程/编排任务）
```

## 二、程序性知识（怎么做）

### 提炼骨架

```
目的 → 前置条件 → 步骤 1..N → 每步输入/输出 → 变异场景
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 目的 | 这个流程要达到什么结果 | 是 |
| 前置条件 | 开始前必须满足的条件 | 是 |
| 步骤 | 按顺序的 1..N 步，每步含：输入、输出、执行方式 | 是 |
| 变异场景 | 常见变异及对应的步骤调整 | 否 |
| 完成标准 | 怎样算「做完了」 | 是 |

### 完整示例

**原始知识**：绘制 BPMN 四步法——套用模板获取符号库 → 拖拽合适符号到画布 → 按流程方向和元素关系连线 → 标注关键元素信息。

**提炼产出**：

```markdown
# CP-xxx：BPMN 绘制四步法

## 目的
从零到一完成一张符合 BPMN 规范的流程图。

## 前置条件
- 已确定要建模的业务流程（文字描述或口头说明即可）
- 有 BPMN 符号库（可用模板导入）

## 步骤

| # | 步骤 | 输入 | 输出 | 执行方式 |
|:---|:---|:---|:---|:---|
| 1 | 套用模板获取符号库 | BPMN 模板文件 | 可用符号库 | 直接导入 |
| 2 | 拖拽合适符号到画布 | 流程的文字描述 + 符号库 | 画布上的元素 | 按流程顺序逐个放置 |
| 3 | 按流程方向和元素关系连线 | 画布上的元素 | 连接完成的流程图 | 实线=顺序流，虚线=消息流，点线=关联 |
| 4 | 标注关键元素信息 | 连接完成的图 | 标注完成的流程图 | 活动名、网关条件、事件名 |

## 变异场景
- 若流程涉及多方参与者：在步骤 2 前加入泳道划分（池+道）
- 若流程已有 UML 流程图作为草稿：步骤 1-2 为「将现有元素映射到 BPMN 符号」

## 完成标准
- 每个流程和子流程都有开始和结束事件
- 无孤立元素（所有元素被连线连接）
- 跨池通信使用消息流（非顺序流）
```

## 三、框架性知识（如何思考/分析）

### 提炼骨架

```
分析目标 → 维度 1..N（定义 + 值域 + 典型示例）→ 维度间关系 → 应用场景
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 分析目标 | 这个框架用来分析什么 | 是 |
| 维度 | 每个分析维度的定义、值域、典型示例 | 是（至少 2 个维度） |
| 维度间关系 | 维度之间是并列、递进还是交叉 | 否 |
| 应用场景 | 什么情况下用这个框架（至少 3 个） | 是 |
| 边界 | 什么情况下不适合用这个框架 | 是 |

### 完整示例

**原始知识**：商业模式画布从客户、价值、渠道、关系、收入、资源、活动、合作、成本九个维度描述商业模式。

**提炼产出**：（此处为完整示例，因篇幅省略部分维度）

```markdown
# CP-xxx：商业模式画布

## 分析目标
系统性描述、设计和分析一个商业模式的完整面貌。

## 维度

| # | 维度 | 定义 | 核心问题 | 值域示例 |
|:---|:---|:---|:---|:---|
| 1 | 客户细分 | 为谁创造价值 | 目标客户是谁？ | B2C 个人用户 / B2B 企业客户 / 平台双边用户 |
| 2 | 价值主张 | 解决什么痛点 | 提供什么价值？ | 省钱 / 省时 / 新体验 / 身份认同 |
| 3 | 渠道通路 | 怎么触达客户 | 认知→购买→交付→售后 | 线上广告 / 线下门店 / API / 合作伙伴 |
| 4 | 客户关系 | 维系什么关系 | 获客/留存/增销 | 自助服务 / 专属客户经理 / 社区 |
| 5 | 收入来源 | 怎么赚钱 | 客户愿意为什么付费？ | 一次性交易 / 订阅 / 授权 / 广告 / 佣金 |
| 6 | 核心资源 | 需要什么资产 | 人财物 + 知识 | 专利 / 品牌 / 供应链 / 数据 |
| 7 | 关键业务 | 做什么最重要 | 生产/方案/平台？ | 制造 / 咨询 / 平台运营 |
| 8 | 重要合作 | 依赖谁 | 供应商和合作伙伴 | 原料供应商 / 渠道伙伴 / 技术联盟 |
| 9 | 成本结构 | 花在哪里 | 固定 vs 可变 | 人力 / 服务器 / 营销 / 采购 |

## 维度间关系
- 1→2：客户细分决定价值主张的针对性
- 2+3→5：价值主张 + 渠道 → 收入模式
- 4→1：客户关系影响客户细分的稳定性

## 应用场景
- 创业初期，为新业务做商业模式设计
- 分析竞争对手的商业模式，找出差异化突破口
- 现有业务转型时，重新审视九个模块是否匹配新战略

## 边界
- 不适用于极度不确定的蓝海探索阶段（此时先做 PMF 验证再填充画布）
- 是静态快照工具，无法描述竞争动态
```

## 四、策略性知识（如何选择）

### 提炼骨架

```
决策问题 → 选项 1..N → 每个选项的适用条件 → 排除规则
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 决策问题 | 要做什么选择（场景化描述） | 是 |
| 选项 | 候选方案 + 每个方案的核心特征 | 是（至少 2 个） |
| 适用条件 | 每个选项在什么情况下最合适 | 是 |
| 排除规则 | 什么情况下明确不用某个选项 | 否 |
| 决策提示 | 帮助做出选择的辅助问题 | 否 |

### 完整示例

**原始知识**：点子过滤器通过内部·能力、内部·意愿、外部·价值、外部·成本四个维度判断产品概念是否值得投入。

**提炼产出**：

```markdown
# CP-xxx：点子过滤器（产品概念筛选）

## 决策问题
在早期阶段判断一个产品点子是否值得投入资源，避免冲动开发。

## 选项
| 结果 | 含义 |
|:---|:---|
| ✅ 通过 | 四个维度均满足条件，值得投入 |
| ⚠ 条件通过 | 大部分满足但有疑虑，需补充验证后决定 |
| ❌ 不通过 | 有关键维度不满足，不应投入 |

## 四个评估维度

| 维度 | 核心问题 | 通过条件 |
|:---|:---|:---|
| 内部·能力 | 团队有无技术高手？资金充足？数据积累？ | 核心能力可自建或可获取 |
| 内部·意愿 | 符合公司使命/愿景/价值观吗？ | 与长期战略一致 |
| 外部·价值 | 赛道够宽吗？用户价值高吗？ | 市场天花板足够 + 用户有真实需求 |
| 外部·成本 | PESTEL 有利吗？行业竞争激烈吗？ | 环境有利或至少不恶劣 |

## 排除规则
- 内部·意愿 = 否 → 直接不通过（与长期战略冲突，不可妥协）
- 外部·价值 = 低 → 直接不通过（没有用户愿意买单，其他都白搭）

## 决策提示
- 「势」是关键：顺势的点子容易通过；逆势的很难
- 深度定制是双刃剑：可能成为壁垒，也可能变成牢笼
```

## 五、经验性知识（踩过的坑）

### 提炼骨架

```
场景 → 错误表现 → 根因 → 纠正 → 预防
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 场景 | 什么情况下容易掉进这个坑 | 是 |
| 错误表现 | 怎么看出来错了；有什么典型症状 | 是 |
| 根因 | 为什么会犯这个错（认知偏差/习惯/信息缺失） | 是 |
| 纠正 | 正确做法是什么 | 是 |
| 预防 | 如何避免下次再犯 | 否 |

### 完整示例

**原始知识**：BPMN 活动三大常见错误——用复杂度决定任务/子流程（正确标准是能否进一步分解）、混淆循环与多实例、发送/接收任务中夹杂执行任务。

**提炼产出**：

```markdown
# CP-xxx：BPMN 活动常见错误

## EXP-1：任务/子流程误判

### 场景
画 BPMN 时判断一个活动应该用「任务」还是「子流程」。

### 错误表现
用复杂度决定：「这个逻辑比较复杂 → 用子流程」/「这个很简单 → 用任务」

### 根因
区分标准是「能否进一步分解为子元素」，而非「复杂度」。简单活动也可以建模为子流程如果建模者决定对其展开。

### 纠正
- 能进一步分解为子元素 → 子流程（Sub-Process）
- 不能进一步分解 → 任务（Task）
- 复杂度不是判断标准，分解能力才是

### 预防
每当你用「复杂/简单」做判断时，停下来问自己：「这个活动在更细的粒度上还能画出一张流程图吗？」

---

## EXP-2：循环与多实例混淆

### 场景
一个活动需要重复执行多次。

### 错误表现
用循环标记处理不同数据集的多次执行。

### 根因
循环 = 同一数据集重复执行直到条件不满足；多实例 = 不同数据集各执行一次。两者语义不同。

### 纠正
- 同一数据反复处理 → 循环标记
- 不同数据各执行一次（如批改 N 份不同作业）→ 多实例标记

### 预防
问自己：「每次执行处理的是同一条数据还是不同的数据？」

---

## EXP-3：发送/接收任务中夹杂执行任务

### 场景
需要向外部发送消息，同时夹杂了业务处理逻辑。

### 错误表现
一个活动既「准备答案」又「发送答案」。

### 根因
发送任务只能发送消息，接收任务只能等待消息。不能在一个任务符号中混合消息交互与业务逻辑。

### 纠正
拆分为两个独立活动：
1. 「准备答案」→ 用户任务（业务逻辑）
2. 「发送答案」→ 发送任务（消息交互）

### 预防
每当你看到发送/接收任务标记时，检查该活动是否只做「发送」或「接收」一件事。
```

## 六、规则性知识（约束条件）

### 提炼骨架

```
约束对象 → 约束内容 → 约束级别 → 违反后果 → 例外
```

### 字段说明

| 字段 | 含义 | 必填 |
|:---|:---|:---|
| 约束对象 | 约束作用于谁（角色/操作/系统） | 是 |
| 约束内容 | 具体约束什么（行为/格式/权限） | 是 |
| 约束级别 | MUST（硬性）/ SHOULD（推荐）/ MAY（可选） | 是 |
| 违反后果 | 违反后会怎样 | 否 |
| 例外 | 什么情况下允许违反 | 否 |

### 完整示例

**原始知识**：整洁架构的四层依赖规则——外层可依赖内层，内层不可依赖外层。

**提炼产出**：

```markdown
# CP-xxx：整洁架构依赖方向约束

## 约束对象
所有架构分层（实体层 / 用例层 / 接口适配器层 / 框架与驱动层）

## 约束内容
- 依赖方向必须从外向内（外层 → 内层）
- 内层不得引用外层的任何符号（类、函数、类型、变量）

## 约束级别
MUST（硬性约束，不可违反）

## 违反后果
- 内层引用外层 → 架构退化，可测试性丧失，替换框架/数据库的成本剧增
- 循环依赖 → 编译/运行时可能正常，但架构腐化不可逆

## 例外
无。这是架构的宪法性约束，不存在例外场景。
```

## 九、视角完备度（可选推荐）

> **定位**：不强制每份协议填满四视角，而是提供一个「多角度看同一知识」的思维路径。`null` 是合法值——代表「此层对该知识域无意义」。

### 四视角定义

| # | 视角 | 核心问题 | 典型适用域 | 不适用范围 |
|:---|:---|:---|:---|:---|
| `functional` | 功能/目的 | 该知识解决什么适应性问题？为什么需要它？| 所有知识 | — |
| `algorithmic` | 算法/流程 | 该知识的加工流程/机制/步骤是什么？ | 所有知识 | — |
| `neural` | 物质基础 | 涉及哪些神经/物理/计算基础？ | 认知心理学、AI、生物 | 商业模式、管理方法论（结构性不适用） |
| `developmental` | 发展变化 | 该知识在个体/组织发展中如何变化？成人前不出现？| 认知、教育、组织 | 数学逻辑、物理定律（恒定不变） |

### 填写格式（可选，协议 frontmatter 底部）

```yaml
perspectives:
  functional: "<一句话 — 该知识解决什么适应性问题> 或 null"
  algorithmic: "<一句话 — 该知识的加工流程/机制是什么> 或 null"
  neural: "<一句话 — 涉及哪些神经/物理基础> 或 null"
  developmental: "<一句话 — 该知识在发展中如何变化> 或 null"
```

### 填写指南

- **不强制四层都填**。`null` 表示「此层结构性不适用」（如商业模式无神经基础），不是分析偷懒。
- **区分结构性缺失与分析偷懒**：若该域应有此视角但你未填 → 分析偷懒，应补充；若该域根本不涉及此维度 → 结构性缺失，填 `null`。
- **发展视角的优先适用域**：认知能力、语言、概念形成、道德推理、组织管理。不适用域：数学定理、物理定律、逻辑规则。

## 十、理解前提条件（Cognitive Prerequisite）

> **定位**：维果茨基最近发展区（ZPD）的知识脑映射——不是"Agent 的学习阶段"（Agent 永远形式运算），而是"理解此协议前，Agent 需要先建立哪些概念锚"。不理解的协议被激活 = 无效激活——信息到了但无法消化。

### 10.1 字段定义

在协议 frontmatter 中增加可选字段：

```yaml
cognitive_prerequisite:
  prerequisite_concepts: ["BPMN", "BPMN.event"]  # 理解此协议需要先建立的概念锚
  auto_derived: ["BPMN"]                          # 自动从 concept-tree parent 链推导的（Agent 生成，不需人工）
  agent_added: ["BPMN.event"]                     # Agent 跨域补标的（检测到协议引用了其他域的概念）
  fallback_path:                                  # ZPD 降级路径——当前置概念缺失时，先看什么
    - { concept: "BPMN", protocol: "CP-027", time_estimate: "2min" }
    - { concept: "BPMN.event", protocol: "CP-022", time_estimate: "3min" }
    - { then: "回到本协议，理解深度将达到  80%" }
```

### 10.2 自动推导规则

```
Agent 在 Step 4 落盘时：
  1. 取协议的 concept_anchor（如 "BPMN.activity.task"）
  2. 沿 concept-tree.json 的 parent 链向上爬：
     "BPMN.activity.task" → parent: "BPMN.activity" → parent: "BPMN"
  3. 对每个父节点，检查 concept-tree 中是否有 ≥1 份协议：
     若有 → 填入 fallback_path（推荐最短路径：concept → protocol）
     若无 → 填入概念树节点 description 做降级说明
  4. 填入 auto_derived 标识自动推导的来源
```

### 10.3 Agent 跨域补标规则

```
Agent 在提炼时检测：
  1. 协议正文中是否出现其他域的概念锚格式文本（"DDD.bounded_context"、"SWOT"）？
  2. 协议正文中是否引用其他域的核心术语（在 concept-tree 其他节点命中）？
  3. 若检测到 → 将跨域 concept_anchor 补入 agent_added，并尝试填充 fallback_path
```

### 10.4 fallback_path 降级行为

| 条件 | 降级动作 |
|:---|:---|
| 前置概念在 concept-tree 有节点且 ≥1 份协议覆盖 | fallback_path 填入最短路径 ✅ |
| 前置概念仅在树中是空节点（无协议） | fallback_path 只填入概念树节点描述，标记 🟡 空缺 |
| 无前置概念（基石协议，如 `prerequisite_concepts: []`） | fallback_path 为 null，标记"此概念为领域基石" |

### 10.5 填写时机

- **自动推导**：协议创建时（Step 4 路由落盘），Agent 当场生成。无需人类介入。
- **跨域补标**：同上——Agent 在提炼时检测跨域引用，自动补入 `agent_added`。
- **后续更新**：若概念树 parent 层级发生变化（如某概念锚被重新 parent），synthesizer.md P3 合成时提示更新受影响协议的 `prerequisite_concepts`。

## 十二、提炼后评断验证（P1）

> 目的：提炼完成不等于正确。在协议落盘前，对提炼产物做轻量验证，防止错误知识固化为「未验证协议」后被当成事实引用。

### 12.1 验证步骤

提炼产物产出后、写入 `protocols/CP-xxx.md` 前，执行以下验证：

#### Step V1：LLM 自检（必做，同步）

用大模型自身知识对提炼产物做交叉检查：

| 检查项 | 问题 | 不合格标记 |
|:---|:---|:---|
| 事实一致性 | 提炼骨架中的定义、关系、边界与大模型已知知识一致吗？ | `judgment: “质疑 — <具体疑点>”` |
| 逻辑自洽 | 提炼产物内部要素之间是否自相矛盾？ | `judgment: “矛盾 — <冲突描述>”` |
| 兜底归因 | 提炼产物未触及大模型已知盲区的知识类型（如实时数据、内部制度）？ | `judgment: “可信”` |

#### Step V2：网搜交叉验证（可选，异步非阻塞）

触发条件（满足任一条即触发网搜）：
- V1 中 `judgment` 为 `“质疑”` 或 `“矛盾”`
- 知识来源为 `public` 且 `validated_count: 0`（首次遇到该领域的公开知识）
- 用户明确要求验证

网搜结果写入验证注释段，不修改协议正文。若网搜结果与提炼产物一致 → `judgment` 升级为 `“可信+网搜一致”`；若不一致 → 保持 `“质疑”` 并附网搜摘要。

#### Step V3：评断注释段格式

每条协议 frontmatter 中增加 `judgment` 注释段（不替代协议正文，独立于正文之外）：

```yaml
judgment:
  verdict: "可信"                  # 可信 / 质疑 — <疑点> / 矛盾 — <冲突描述>
  self_check: "LLM 自检通过，定义与已知 BPMN 标准一致"
  web_check: null                  # 或 "2026-05-17 网搜一致: [关键词]"
  doubts_resolved: ["子流程与调用活动的边界"]  # 从 P0 诠释自检的 doubts 中已澄清的项
  note: null                       # 保留的已知局限或边界说明
```

### 12.2 验证结果对协议状态的影响

| verdict | 协议 `status` | `validated_count` | 使用时标注 |
|:---|:---|:---|:---|
| `“可信”` | candidate → 保持不变 | +1 | 无特殊标注 |
| `“可信+网搜一致”` | candidate → active（直接升级） | +2 | 无特殊标注 |
| `“质疑”` | candidate（待修订） | 0 | 运行时前置 `⚠ 待验证` |
| `“矛盾”` | candidate（待修订） | 0 | 运行时前置 `⚠ 存疑 — <冲突描述>` |

### 12.3 与 P0 诠释自检的衔接

P0（`classifier.md` Step 1.5）中的 `doubts` 字段传入此处，若提炼过程中解答了困惑 → 填入 `judgment.doubts_resolved`；若仍未解答 → 填入 `judgment.note` 并标注 `“理解尚不完全”`。

## 十一、认知深度自评（P1 扩展）

> **定位**：P1 评断验证只管「对不对」，认知深度自评管「有多深」。同一份事实正确的协议，深度可能天差地别——深度决定 Agent 激活后的行为模式。

### 11.1 七级认知深度

| 级别 | 名称 | 判别信号（来自 Chi/Feltovich/Glaser/Ericsson 专家-新手文献） | Agent 行为模式 |
|:---:|:---|:---|:---|
| **L1** | 标签级 | 仅给出概念名 + 一句话定义 | **模式匹配**——只知道"这是什么"，无法推导，遇到边界案例即兴发挥 |
| **L2** | 特征列举级 | 列举了特征/类型/维度，但无关系说明 | **清单对照**——逐一贴标签。简单流程能过，边界案例卡住并逐条查规范 |
| **L3** | 关系级 | 标注了概念间的因果/层级/对比关系 | **结构化思考**——知道子流程≠任务并能说清为什么。但不会在新场景中主动变通 |
| **L4** | 原理级 | 揭示了"为什么"——该知识解决什么底层问题，提供了可推导的原则 | **原理推导**——用原则而非清单做决策。新场景中自洽应用，不需要逐条查规范 |
| **L5** | 边界级 | 明确标注了适用范围和不适用范围，包括"此场景不该用我" | **精确边界——不仅知道何时用，还知道何时不用。能区分"该用 BPMN 还是 DMN/CMMN" |
| **L6** | 跨域桥接级 | 连接了两个以上概念域，形成跨域融汇（如 BPMN 泳道 = DDD 限界上下文） | **跨域综合——在画流程时同步思考系统集成边界。产出方案深度超过用户初始预期 |
| **L7** | 元认知级 | 协议自身标注了"我的局限是什么""我在什么条件下需要被修订" | **自我监测——知道自己的知识什么时候会失效。不会在过时场景中信誓旦旦给错误方案 |

### 11.2 判别示例（同一概念，不同深度）

以 "BPMN 活动类型" 为例，展示 L1-L7 的同一概念在不同深度下的提炼差异：

| 级别 | 对应提炼 |
|:---|:---|
| L1 | "BPMN 活动（Activity）是流程中执行的工作单元" |
| L2 | L1 + "活动分三类：任务（原子）、子流程（可展开）、编排任务（跨参与者消息交换）。任务又有服务/用户/手动/业务规则/脚本/发送/接收七种" |
| L3 | L2 + "区分任务和子流程不看复杂度看可分解性。区分发送任务和接收任务看消息方向（发出 vs 等待）" |
| L4 | L3 + "BPMN 活动分类的底层原则：看'谁在什么粒度上执行了多少动作'。单一参与者×单一时刻×单一动作→任务；可继续展开为子流程→子流程；跨参与者消息交换→编排任务" |
| L5 | L4 + "本分类体系适用于结构化业务流程。不适用：探索性创新流程（此时应使用 DMN+CMMN），高度并行的无角色分工流程" |
| L6 | L5 + "BPMN 泳道间的消息流 = DDD 限界上下文间的集成事件。画 BPMN 发送/接收任务时，同步定义 API 契约的幂等性、超时、重试策略" |
| L7 | L6 + "本协议的修订触发条件：当出现新的交互范式（如智能合约自动执行——它既是外部系统又是执行者）时，'服务任务 vs 发送任务'的区分可能部分失效" |

### 11.3 填写格式

在协议 frontmatter 中增加：

```yaml
depth_level: 4                    # L1-L7，提炼时 Agent 自评
depth_note: "提供了可推导的原则，但未覆盖边界条件"  # 自评依据，可选
```

### 11.4 联动规则

- **L3 以下协议**：在 Step R 读后总结中自动标记 `⚠ 待深化`，提示"该协议停留在特征列举/标签级，未提供原理推导或关系说明"
- **同域深度分布**：`synthesizer.md` 动作 6「学习路径推荐」优先推荐 L4+ 协议为消化终点，L1-L2 协议为消化起点
- **激活优先级**：同 `concept_anchor` 下 L4+ 协议在过滤器中排序优于 L1-L3（同等验证状态下）
- **与视角完备度的联动**：L1-L2 协议 + `perspectives.functional = null` → 高度可疑（功能视角是最基础的，L1 丢了说明提炼严重不足）
