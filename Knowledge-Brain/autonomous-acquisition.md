# 自主获取回路

## 一、三种获取模式

| 模式 | 触发 | 入口 | 归口 |
|:---|:---|:---|:---|
| **投喂型（Feed）** | 用户直接提供知识源："学习这篇/这本书" | `gateway-command-router.mdc`「学习」→ 消化管线 | 消化管线 → Step R → Step S |
| **任务驱动型（Task-Driven）** | `task-init-protocol.mdc` Step 2-KB 检测 K2/K3 | Step 2-KB > 认知需求分析 > 检索 | 待消化池 → 任务完成 → 消化管线 |
| **探索驱动型（Explore-Driven）** | 用户开放指令："收集 XX 领域的主流思维模型" | 「学习」无源分支 > 意图澄清 > L1/L2/L3 | 待消化池 → 用户结束探索 → 消化管线 |

**归口统一原则**：所有来源的知识进入知识脑，必须经过消化管线的 Step R（反思）和 Step S（扫描）。怎么来的不管，两道门必须过。

## 二、意图澄清层（Intent Clarification）

### 2.1 任务驱动型的意图澄清

**触发**：Step 2-KB 检测到 K2/K3

**过程**：对每个子任务提取认知需求类型：

| 认知需求 | 子任务信号 | 缺什么 |
|:---|:---|:---|
| 分类/结构化 | 需要把一堆东西归类 | 分类框架 |
| 优先级/决策 | 需要在多个选项中选 | 决策矩阵 |
| 评估/判断 | 需要判断好坏对错 | 评估标准 |
| 设计/创建 | 需要从零搭一个东西 | 设计方法论 |
| 诊断/归因 | 需要找到根因 | 诊断方法 |
| 预测/假设 | 需要推测未来发展 | 预测模型 |

**输出**：缺口类型列表，附带优先级。不自动搜索——仅标注缺口，在 Step 3 提案末尾追加：
```
💡 知识脑缺口检测：<认知需求>
   → 是否开启知识搜索？
     [是，搜索] [否，直接执行]
```

### 2.2 探索驱动型的意图澄清

**触发**：用户开放指令，如"收集主流思维模型"、"学习所有战略框架"

**过程**：Agent 将开放指令拆为结构化分类 + 澄清标准：

**步骤 1：分类拆解**

```
你让我收集 <开放目标>。我先拆成几个分类，你看是否认可：

1. <分类1>（如：决策与评估模型）
2. <分类2>（如：问题分析与诊断模型）
3. ...

确认分类后，我开始按类别检索。
```

**步骤 2：标准澄清**

```
另外，关于 <关键判断标准>：
  [A] <标准A>
  [B] <标准B>
  [C] <标准C>
```

**步骤 3：确认后进入 L1 编目**

### 2.3 意图澄清的通用规则

- **不靠大模型凭空分类**——引用已知分类体系作为锚（如"管理思想史通常将 X 分为…"）
- **分类上限**：每轮 ≤6 个类（防止编出过于琐碎的分类）
- **未归类即红旗**：如果候选框架无法归入任何确认的分类 → 标注"跨类/分类体系外"，降权处理

## 三、检索与验证层（Retrieval & Verification）

### 3.1 分层检索（探索驱动型专用）

| 步骤 | 动作 | 产出 | 验证 |
|:---|:---|:---|:---|
| **L1 编目** | 大模型按确认的分类体系，列出每类下已知的框架名（仅出框架名，不出原理） | 分类目录草稿 | 你确认/调整分类 |
| **L2 深挖** | 对确认的分类，大模型逐一列出框架名 + 一句话核心原理 + 记忆依据标签 | 按类组织的框架清单 | G1 来源验证 |
| **L3 入库** | 你确认清单 → 进入消化管线 | 协议落盘 | P2 四问闸门 + Step R + Step S |

**L2 深挖的记忆依据标签**：
- 大模型必须标注「记忆依据」——我知道这个框架是基于什么：
  - `training_data`：我在训练数据中见过，有明确的出处记忆
  - `unsure`：我推断这个领域应该有这样一个框架，但我不确定具体名称和出处
- 仅 `training_data` 标签的候选进入 G1 验证。`unsure` 的直接丢弃。

**L2 深挖的搜索提示词**：
```
在 <领域> 的 <分类> 中，有哪些被广泛认可的思维框架？
对每个框架：① 一句话核心原理 ② 出处（谁提出的/哪本书/哪篇论文）
只输出你确定存在的框架。不确定的不要列。
```

### 3.2 任务驱动型检索

目标明确、范围窄 → 大模型直接列候选框架 → G1 验证 → Discovery Queue。

### 3.3 G1 来源验证（五级评级 —— 核心防御）

对每个候选框架，网络交叉验证后评级：

| 级别 | 来源 | 可信度 | 入库动作 |
|:---|:---|:---|:---|
| **S** | 学术论文（Google Scholar 可查）、正式标准文档（ISO/OMG）、权威教材（多所高校采用） | 极高 | ✅ 强验证，直接入库，初始置信度 ⚠ 需关注 |
| **A** | Wikipedia 独立条目且无争议标签、Harvard Business Review / MIT Sloan 等权威期刊 | 高 | ⚠ 需关注，正常入库 |
| **B** | 知名咨询公司白皮书（McKinsey/BCG/Bain）、行业标准组织的官方出版物 | 中 | 🔶 未验证，标注"商业来源"，入库 |
| **C** | 个人博客、Medium、知乎专栏、无名出版社的书 | 低 | 🔶 未验证 + 标注"弱来源"，入库但降权 |
| **F** | 网络完全找不到独立来源 | 极低 | 标注"⚠ 可能为大模型合成"，**不入库** |

**额外检测**：对 S/A 级来源，还需搜索是否已有被批评/证伪的记录（如"BCG Matrix criticism"）。

### 3.4 Discovery Queue 产出

每条候选框架一张摘要卡：

```
📋 BCG 矩阵
  认知需求：分类/评估 — 产品组合决策
  核心原理：按市场增速和相对市场份额将产品分为四象限
  出处：Bruce Henderson, Boston Consulting Group, 1970
  G1 评级：A — Wikipedia 独立条目 + HBR 原文可查
  潜在用途：评估多个业务的投资优先级时使用
  竞争框架：GE 矩阵（多因素版 BCG）— 更细粒度但更复杂
  批评记录：有 — 被批评为过度简化、只关注两个维度
```

### 3.5 对投喂型知识的反向验证

你投喂的书/文章也走 G1 交叉验证：
- 网络搜索"<书中的核心主张> + criticism" → 是否有被证伪或争议的记录
- 网络搜索"<书中的核心主张> + alternative" → 是否有竞争理论
- 检测结果标注在协议的 `cross_check` 字段

## 四、自标注与自动入库

### 4.1 自主获取的完整自标注字段

每份协议入库时自动标注：

```yaml
verification:
  source_type: "llm+web_verified"
  source_details: "Wikipedia + HBR 1992"
  g1_rating: "A"
  memory_tag: "training_data"   # 来自 L2 深挖标注
  cross_check:
    - against_protocol: null     # 暂无同域已有协议（首次入库为空）
      relation: null
    - against_web: "GE Matrix 被认为是 BCG 的改进版"
    - criticism: "被批评为过度简化"
  confidence_at_entry: "⚠ 需关注"
```

### 4.2 入库规则

- **S/A/B/C 级**：自动入库——不弹 AskQuestion。仅在对话中回执一句：
  ```
  📋 自主获取：BCG 矩阵已入库（⚠ 需关注，A级源：Wikipedia + HBR 1970）。
      标记批评记录：过度简化。等待实战验证。
  ```
- **F 级**：不入库。在 Discovery Queue 中标注 `不入库 — 网络无独立来源`。
- **你随时可介入**："CP-xxx 这条我不信，去掉" → 标记 `falsified`。
- **你随时可降权**："CP-xxx 暂时别用" → 置信度 → `⛔ 已降权`。

## 五、待消化池（Pending Digestion Queue）

### 5.1 定位

`Knowledge-Brain/pending-digestion.json` — 一个极简 JSON 暂存文件。任务型和探索型的 Discovery Queue 产出暂存于此。

### 5.2 结构

```json
{
  "source": "task-T005",
  "created": "2026-05-18T15:00:00",
  "candidates": [
    {
      "candidate_id": "DISC-001",
      "title": "BCG 矩阵",
      "cognitive_need": "分类/评估",
      "g1_rating": "A",
      "core_claim": "按市场增速和相对市场份额将产品分为四象限",
      "source_details": "Wikipedia + HBR 1970",
      "acquired_at": "2026-05-18T15:00:00"
    }
  ]
}
```

### 5.3 触发消化

- **任务驱动型**：任务完成后，收束步骤 `flow-behavior-auto-receipt.mdc` §C 检测 `pending-digestion.json` 非空 → 逐条走消化管线
- **探索驱动型**：探索结束后（用户说"可以了"或结束本轮话题），同上触发

## 六、归口统一

```
投喂型 ────────────────────→ 消化管线 ──→ Step R ──→ Step S ──→ 协议落盘
                                    ↑
任务驱动型 → 认知需求 → 检索 → 待消化池 → 收束触发消化
                                    ↑
探索驱动型 → 意图澄清 → L1→L2→L3 → 待消化池 → 结束触发消化
```

所有来源共享以下步骤：
- **P2 四问闸门**：自述问题、边界问题过滤逻辑不清的假框架
- **Step R 读后总结**：Agent 主观反思有用/无用、K↑ 升级
- **Step S 全域巡检**：新入库协议与已有系统的碰撞检测
- **P3 互证**：同域累积 ≥3 条后触发新旧互证

## 八、你保留的权力

| 行动 | 触发方式 | 效果 |
|:---|:---|:---|
| 踢出 | "CP-xxx 这条我不信，去掉" | 标记 `falsified`，不再激活 |
| 降权 | "CP-xxx 暂时别用" | 置信度 → `⛔ 已降权` |
| 加速验证 | 使用后觉得好 → 标记 `effective` | 置信度快速升级 |
| 打标签 | "CP-xxx 补充一下出处是…" | 更新 `verification` 字段 |
| 什么都不做（默认） | — | 知识自治——互证 + 实战自然筛选 |
