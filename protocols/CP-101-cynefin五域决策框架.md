# Cynefin 五域决策框架

## 核心概念

Cynefin（威尔士语，意为"你多重归属的地方"）是 Dave Snowden 在 1990 年代末创建的决策支持框架。它帮助领导者理解问题的本质——**在行动之前，先判断问题属于哪个域**。

每种域有不同的因果结构 → 不同的应对策略。**域错了，再好的方法也反效果**。

---

## 五域详解

### 1. 清晰域（Clear / Simple）— "照着做就行"

```
因果：明确、可重复、一眼便知
策略：感知 → 分类 → 响应（Sense → Categorize → Respond）
方法："最佳实践"（Best Practice）
```

| 特征 | 说明 |
|:---|:---|
| **典型场景** | 标准化流程、成熟技术选型、照菜谱做菜、常规审批 |
| **决策** | 存在唯一正确答案或明确规则。不需要创新——**遵从而非思考** |
| **风险** | 过度简化——把复杂问题误判为清晰问题。固守"最佳实践"当环境变了 |
| **示例** | "用 AWS S3 存静态文件"——明确定式，照做即可 |

**危险信号**：你在清晰域待得太舒服了——任何看起来简单的问题都值得再检查一遍，确认没有隐藏的复杂性。

---

### 2. 繁杂域（Complicated）— "需要专家分析"

```
因果：存在，但需要专业知识才能发现。可能有多个正确答案
策略：感知 → 分析 → 响应（Sense → Analyze → Respond）
方法："好实践"（Good Practice）——不止一个正确做法
```

| 特征 | 说明 |
|:---|:---|
| **典型场景** | 架构设计、技术选型对比、医疗诊断、法律分析 |
| **决策** | 需要专家判断。先收集和分析信息，再选择方案。**分析 > 直觉** |
| **风险** | 专家过度自信（"我见过这个，就是 X"——但实际上是 Y 的变体）。不同专家给出矛盾答案时停滞不前 |
| **示例** | "我们的系统应该用单体还是微服务？"——需要专家分析上下文，两个答案都可能正确 |

**关键区分**：繁杂域有"正确答案的集合"，只是你不知道哪个是最优的——需要专家的分析。复杂域没有"正确答案"——答案会涌现。

---

### 3. 复杂域（Complex）— "试了才知道"

```
因果：只能在事后追溯，无法事前预测。模式随时间涌现
策略：探测 → 感知 → 响应（Probe → Sense → Respond）
方法："涌现实践"（Emergent Practice）——实验驱动
```

| 特征 | 说明 |
|:---|:---|
| **典型场景** | 新产品市场验证、团队文化建设、育儿、商业模式探索 |
| **决策** | 无法通过分析找到"正确答案"——答案在互动中涌现。**先做一个小实验，观察结果，调整方向** |
| **风险** | 试图用繁杂域的专家分析法预测不可预测的结果（"我们雇个顾问来告诉你该怎么做"——顾问也不知道） |
| **示例** | "这个新功能用户会喜欢吗？"——不知道。做一个最小版本，投放给 100 个用户，看反馈 |

**核心策略**：在复杂域，不做大计划。做**安全失败（safe-to-fail）实验**——低成本、小范围、可逆的探测试验。

---

### 4. 混沌域（Chaotic）— "先稳住再理解"

```
因果：不可辨别。系统正在瓦解
策略：行动 → 感知 → 响应（Act → Sense → Respond）
方法："创新实践"（Novel Practice）——紧急干预
```

| 特征 | 说明 |
|:---|:---|
| **典型场景** | 生产环境宕机、公关危机、自然灾害响应 |
| **决策** | **先行动，再理解**。首要目标是稳定局势——稳定后问题往往会降级到其他域 |
| **风险** | 试图在行动前先分析——会错过干预窗口。行动后不反思——停留在混沌状态 |
| **示例** | "服务器被 DDOS 攻击了"——立即切流量/限速/报警。稳定后再复盘"怎么会发生" |

**关键原则**：混沌域中，领导者的角色是**采取果断行动以稳定局势**，然后尽快将问题转移到复杂域或繁杂域进行解决。

---

### 5. 失序域（Disorder）— "我不知道在哪"

```
状态：不知道当前处于哪个域。不同部分可能在不同域
策略：拆解 → 将各部分分配到正确的域 → 分别应对
```

| 特征 | 说明 |
|:---|:---|
| **典型场景** | 涉及多部门的复杂项目、初创公司的混乱期、大规模组织变革 |
| **诊断** | 问题中不同部分可能有不同的因果结构——不能整体判域，要拆解后逐部分判 |
| **风险** | 停留在失序状态不做拆解——所有人按自己习惯的方式应对，产生冲突 |

---

## 域间转换与边界

### 域之间的"悬崖"

```
清晰域 ←→ 繁杂域（相对安全，专家知识可能过时即转入清晰域）
繁杂域 ←→ 复杂域（需要警惕——将复杂问题错判为繁杂是最常见的错误）
复杂域 ←→ 混沌域（危机发生时的自然转换）
```

### 混沌域的"灰色边界"

- **混沌域 → 复杂域**：稳定后模式开始涌现——从"救火"转为"探测"
- **清晰域的陷阱**：在清晰域待得太舒服 → 系统悄悄滑入混沌域（"为什么突然崩溃了？"——因为你没注意到边界被越过了）

---

## 使用流程

1. **先判域**：这个问题的因果结构是什么样的？
   - 一眼知道答案？→ 清晰域
   - 需要专家分析？→ 繁杂域
   - 只能试了才知道？→ 复杂域
   - 系统在崩溃？→ 混沌域
   - 不知道？→ 失序域（拆解后逐部分判）

2. **匹配策略**：用对应域的方法论——不要跨域用错方法

3. **动态监测**：域可能随时间变化——问题解决过程中可能从一个域转移到另一个域

### 最常见的错误

在复杂域用繁杂域的方法：请一群专家开会，试图通过分析预测一个不可预测的市场——浪费时间和金钱。正确做法：做 3 个小实验，看市场怎么反应。
