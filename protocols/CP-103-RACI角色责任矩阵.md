# RACI 角色责任矩阵

## 核心定义

RACI 是项目管理中最广泛使用的责任分配框架。它将每个任务的角色分为四类，解决"谁做什么、谁决定、谁参与、谁知情"这一协作中最常见的混乱源。

| 角色 | 英文 | 含义 | 数量约束 |
|:---|:---|:---|:---|
| **R — 执行者** | Responsible | 实际动手做这件事的人 | ≥1 人，但尽量少（避免沟通开销） |
| **A — 审批人** | Accountable | 最终拍板、对成败负责的人 | **有且只有 1 人** |
| **C — 咨询者** | Consulted | 在决策前需要征求意见的人 | ≥0 人，双向沟通 |
| **I — 知情者** | Informed | 决策后需要被告知结果的人 | ≥0 人，单向通知 |

---

## 四个角色的深层含义

### R — 执行者（Responsible）
- "我动手做"——实际完成任务的执行人
- 可以多人，但每个任务的总执行人数应该**尽量少**——R 越多，沟通和协调开销越大
- R 不是"应该做"而是"被明确分配了要做"

### A — 审批人（Accountable）
- "这事最终是我拍板的"——对任务成败负最终责任的人
- **每个任务有且只有一个 A**——这是 RACI 最核心的规则。多人共担 = 无人真正负责
- A 不一定是最高层领导——A 是"这件事做不好，我承担后果"的人
- A 可以同时是 R（但建议区分——自己做自己审批容易失去质量闸门）

### C — 咨询者（Consulted）
- "做决定之前，先问问我的意见"——提供专业输入的人
- **双向沟通**：执行者主动征求咨询者的意见，咨询者给出反馈
- 典型角色：法务、安全、架构、领域专家
- 注意：咨询 ≠ 否决。C 提供输入，A 做最终决策

### I — 知情者（Informed）
- "做完了告诉我一声，不需要你等我回复"——需要了解进展和结果的人
- **单向通知**：执行者/审批人通知知情者，知情者不需要回应
- 典型角色：相关团队负责人、上下游依赖方、高层管理者
- 不要把太多人放在 I——每个人都被告知 = 没人真正关注

## RACI 矩阵格式

```
        任务1   任务2   任务3   任务4
张三     R       I       C       R
李四     A       R       I       A
王五     C       A       R       C
赵六     I       C       A       I
```

**纵轴**：团队成员（谁）
**横轴**：任务/决策（做什么）
**交叉点**：该人在该任务中的角色（R/A/C/I 之一）

### 常见变体

| 变体 | 额外角色 | 适用场景 |
|:---|:---|:---|
| **RASCI** | S = Support（辅助执行者完成任务） | 大团队，R 需要明确辅助人 |
| **RACI-VS** | V = Verifier（验证产出是否符合标准）+ S = Signatory（最终签字） | 合规要求高的项目 |

## 使用指南

### 创建步骤

1. **列任务**：将项目分解为具体的、有明确交付物的任务。按时间顺序排列
2. **列人员**：列出所有参与方（个人或角色）。注意人数不要太多（>10 人时考虑分组）
3. **逐任务赋角色**：对每个任务，依次确定 A（审批人）→ R（执行者）→ C（咨询者）→ I（知情者）
4. **检查垂直负载**：每个人是否超负荷？某人是大部分任务的 A → 瓶颈风险
5. **检查角色缺失**：每个任务都有 A 吗？（必须）每个任务的 R 合理吗？
6. **团队对齐**：全团队一起过一遍矩阵——让每个人确认自己的角色

### 质量自检清单

| 检查项 | 问题 |
|:---|:---|
| **A 唯一性** | 每个任务只有一个 A 吗？ |
| **A 指认** | 每个 A 知道自己要对这个任务负最终责任吗？ |
| **无真空** | 每个任务都有 R（执行者）吗？ |
| **C 适度** | C 是否太多了？（太多 C = 决策瓶颈） |
| **I 有意义** | I 是否真的需要被通知？（被"通知一切"的人会无视一切） |
| **平衡** | 单个人是否承担了太多的 R 或 A？（瓶颈检测） |

### 常见陷阱与纠正

| 陷阱 | 表现 | 纠正 |
|:---|:---|:---|
| **多个 A** | 两个人都觉得自己能拍板 → 冲突或推诿 | 强制二选一——"如果这件事做砸了，谁承担责任？" |
| **A 太多任务** | 某人是 80% 任务的 A → 实际上不可能对每个任务真正负责 | 向下授权——让更多人有自己的 A 域 |
| **C 太多** | 每个决策前要咨询 8 个人 → 决策速度极慢 | 区分"需要咨询"和"nice to have 的想法"。咨询者应该是有专业判断力能实质改变决策的人 |
| **RACI 写成后不更新** | 角色变了但矩阵不变 → 比没有 RACI 更糟（虚假的清晰度） | 每次人员变动 / 项目阶段转换时更新矩阵 |
| **把 RACI 当惩罚工具** | "你因为这个没做好，看 RACI 上你是 R" | RACI 是**事前对齐**工具，不是事后追责工具 |

## 何时使用 RACI（何时不用）

### 适合 RACI
- 项目涉及 ≥3 个部门/团队
- 责任边界模糊，常出现"我以为你在做"的情况
- 多人需要为同一决策提供输入
- 外部利益相关者需要明确的知情路径
- 项目有明确的阶段和截止日期

### 不适合 RACI
- 小型团队（2-3 人日常协作）——口头对齐更高效
- 流程标准化极高（每人角色固定不变）——RACI 变成不必要的文档
- 快速迭代的敏捷团队——RACI 的静态性质与持续变化的角色冲突
- 单人任务

## 与其他决策工具的配合

| 工具 | 互补关系 |
|:---|:---|
| **Cynefin** | RACI 定"谁来做决定"，Cynefin 定"这个决定应该怎么做出" |
| **决策树** | RACI 分配决策的 A（审批人），决策树提供 A 做决策时使用的分析框架 |
| **RACI + RAPID** | RACI 覆盖日常任务，RAPID（Recommend-Agree-Perform-Input-Decide）覆盖重大决策的额外审批层 |
