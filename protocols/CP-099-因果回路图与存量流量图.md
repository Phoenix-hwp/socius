# 因果回路图与存量流量图

## 因果回路图（Causal Loop Diagram, CLD）

### 基本要素

| 要素 | 规则 | 示例 |
|:---|:---|:---|
| **变量** | 必须是能随时间增减的量，用名词短语表达（如"客户满意度""库存数量"） | ✓ "团队士气" ✗ "士气下降"（这是行为，不是变量） |
| **因果箭头** | 从原因指向结果 | "加班时数 → 代码质量" |
| **正因果关系（+）** | 原因增减与结果增减同向 | "营销投入(+)→ 品牌知名度"（投入越多，知名度越高） |
| **负因果关系（-）** | 原因增减与结果增减反向 | "代码审查(-)→ Bug 数量"（审查越多，Bug 越少） |
| **延迟标记** | 用 `//` 或 `delay` 标注在箭头上 | "招聘 → 到岗（// 2 个月延迟）" |

### 回路极性判断

- **增强回路（R）**：回路中有偶数个负因果箭头（0、2、4...）→ 自我强化，引发增长或崩溃
- **调节回路（B）**：回路中有奇数个负因果箭头（1、3、5...）→ 自我修正，追求稳定或目标

**快速判断法**：如果变量的变化趋势会自我放大 → 增强回路；如果变化趋势会自我收敛 → 调节回路。

### CLD 绘制最佳实践

#### 1. 变量命名
- 用**可增减的名词短语**（数量/水平/程度/质量）
- 避免用动词（动词是行为，不是系统中的量）
- 区分"实际状态"和"感知状态"（如"市场需求" vs "感知市场需求"）

#### 2. 范围界定
- 先确定**时间范围**（关注的时间跨度决定了哪些延迟和趋势是相关的）
- 只纳入**显著影响**变量——问"这个变量翻倍或减半是否会对核心问题产生实质影响？"
- 一个 CLD 聚焦一个主题，不是一个系统的"一切因果关系"

#### 3. 包含延迟
- 物理延迟（如建设周期、运输时间）
- 信息延迟（如报告周期、数据采集滞后）  
- 感知延迟（如消费者习惯变化需要时间）
- **延迟是系统行为的关键驱动因素**——省略延迟会导致"为什么方案不立刻生效"的误解

#### 4. 区分实际 vs 感知
- 真实条件（如"库存数量"）和感知条件（如"报告的库存数量"）是不同的变量
- 感知变量通常有延迟、误差、偏差——这些影响决策质量

#### 5. 命名回路
- 为每个闭合回路命名，描述其动态行为（如"品牌强化回路""倦怠回路"）
- 帮助团队快速理解系统结构的"故事线"

## 存量流量图（Stock and Flow Diagram, SFD）

### 核心概念

| 元素 | 定义 | 特性 | 示例 |
|:---|:---|:---|:---|
| **存量（Stock）** | 系统中的累积量——某一时刻的快照值 | 随时间累积，是系统的"记忆" | 用户数、技术债务、品牌资产 |
| **流量（Flow）** | 改变存量的速率——单位时间内的变化量 | 带时间单位（/天、/月） | 新增用户/天、还债速率/月 |
| **云** | 系统边界的起点/终点 | 不关心的来源或去向 | 新用户从哪里来（外部） |

**关键关系**：存量(t) = 存量(t-1) + 流入(t) - 流出(t)

### CLD → SFD 转换步骤

1. **指定单位**：为所有变量定义单位，迫使思考"时间维度"——哪些变量带时间单位？
2. **识别存量**：寻找系统中的累积量（库存、人数、信任、知识）
3. **识别流量**：寻找改变存量的活动（进货/出货、入职/离职、学习/遗忘）
4. **连接存量与流量**：流量流入或流出存量——这个物理结构决定了系统的动态行为
5. **区分物质流与信息流**：CLD 中两者都用同一种箭头；SFD 中要分开（物质流改变存量，信息流影响流量速率）

### 单位一致性原则

- 流入与流出的存量必须使用兼容的单位
- 例：存量=人数(人)，则流入和流出都必须带"/月"（人/月）
- 违反单位一致性的模型在仿真时会崩溃或产生荒谬结果

### 存量-流量与 CLD 的互补

| 维度 | CLD | SFD |
|:---|:---|:---|
| **目的** | 快速沟通反馈结构 | 精确描述系统行为 |
| **精度** | 定性 | 定量（可仿真） |
| **学习门槛** | 低（30 分钟可上手） | 中（需理解存量-流量区别） |
| **适合阶段** | 问题诊断初期、团队对齐 | 假设验证、策略仿真、敏感性分析 |
| **典型产出** | "哦！原来是这个回路在起刹车作用" | "如果延迟从 2 周变成 4 周，系统的震荡幅度会增加 3 倍" |

## 从问题到模型：三步法

### 第一步：定义核心问题
- 用一句话描述你关心的系统行为：什么变量？什么趋势？什么让你觉得这是"问题"？
- 例："用户数增长曲线为什么在第 6 个月变平？"

### 第二步：识别相关变量与关系
- 圈出 5-10 个核心变量
- 逐对连接因果箭头（+/-）
- 找到闭合回路，判断极性（R 或 B）

### 第三步：测试模型的解释力
- 模型能否解释观察到的行为？
- 模型是否指出了新的杠杆点？
- 与其他人的认知是否一致？（不一致本身就是有价值的信息）

### 常见陷阱

| 陷阱 | 表现 | 补救 |
|:---|:---|:---|
| **变量不可量化** | "公司文化变好了"——无法测量变化量 | 拆解为可观测的代理变量（如"自愿离职率"） |
| **箭头方向混乱** | 因果方向不确定 | 用"如果 X 增加（其他不变），Y 会增加还是减少？"测试 |
| **混淆存量与流量** | 把流量当存量用 | 存量的单位不带时间（"人数"），流量的单位带"/时间"（"入职/月"） |
| **忽略延迟** | 模型预测"立刻生效"但现实是"3 个月后才有反应" | 在关键因果链上标注延迟长度 |
| **过度复杂** | 一个 CLD 有 50+ 变量 | 聚焦核心问题，砍掉"翻倍也不影响核心问题"的变量 |
