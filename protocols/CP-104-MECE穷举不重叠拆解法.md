# CP-104：MECE 穷举不重叠拆解法

## 定义

MECE（Mutually Exclusive, Collectively Exhaustive）是麦肯锡前顾问 Barbara Minto 在 1960 年代末提出的分组原则，是金字塔原理和所有管理咨询问题拆解的第一性原理。

| 维度 | 含义 | 反例 |
|:---|:---|:---|
| **M**utually Exclusive（相互独立） | 各类别之间零重叠。同一项只能落入一个类别 | 客户按"年龄+职业"双重标准分——30岁程序员同时落入两个组 |
| **C**ollectively Exhaustive（完全穷尽） | 所有类别合起来覆盖全部范围，无遗漏 | 问题分析只列"A/B/C"三个原因——漏了"制度本身缺陷" |

## 三类 MECE 拆分方法

### 方法一：数学拆分（最严）

用数学公式穷举——确保分母全覆盖后分子无重叠：

```
利润 = 收入 - 成本 = (价格 × 数量) - (固定成本 + 可变成本)
→ 利润下降的可能原因：
  ├─ 收入端：价格下跌 / 数量减少
  └─ 成本端：固定成本上升 / 可变成本上升
```

### 方法二：流程拆分（适合过程/操作类问题）

按时间轴或因果链拆分，每一步是上一步的必要前提：

```
用户流失问题 → 按漏斗拆：
  曝光 → 点击 → 注册 → 激活 → 留存 → 付费 → 续费
  每层分析流失率 → ME（同一用户不会在两个阶段同时流失）/ CE（所有流失必在某一层）
```

### 方法三：维度拆分（适合分类/定位类问题）

选一个维度做穷举分类，确保该维度的所有取值都覆盖：

| 场景 | 拆分维度 | ME 检查 | CE 检查 |
|:---|:---|:---|:---|
| 客户细分 | 年收入（<10w / 10-30w / 30-100w / >100w） | 区间不重叠 | 覆盖全部收入段 |
| 竞品分析 | 竞争关系（直接/间接/替代/潜在） | 每种关系互斥 | 覆盖所有竞品类型 |
| 代码模块 | 职责层（UI层/业务层/数据层/基础设施层） | 每个类只属于一层 | 架构中所有代码都有归属层 |

## 使用 MECE 的 4 步操作流程

| # | 步骤 | 关键检查 |
|:---|:---|:---|
| 1 | 确定拆分范围 | 我要拆分的"整体"是什么？边界画在哪？ |
| 2 | 选拆分方法 | 用数学/流程/维度三者之一 — 不混用 |
| 3 | 验证 ME | 任取一项：能不能同时属于两个类别？能 → 有重叠，改 |
| 4 | 验证 CE | 任取一个可能的项：有没有归属类别？没有 → 有遗漏，补 |

## 编码场景中的 MECE

| 场景 | MECE 应用 | 反例 |
|:---|:---|:---|
| API 接口设计 | 按资源路径拆分（`/users` `/orders` `/products`）→ 无交集 | `/users/orders` 和 `/orders` 语义重叠 |
| 数据库表设计 | 按实体拆分，每张表一个唯一主键 — 不跨表重复存储 | 订单表和用户表都存 `user_name`（双源不一致） |
| 异常处理 | 按异常类型覆盖：网络/认证/业务/系统 — 每种异常的 catch 块不重叠 | `catch Exception` 兜底导致多个异常走同一条路径 |
| 测试用例 | 等价类划分：有效输入/无效输入/边界值 — 每个测试仅匹配一类 | 同一 bug 可被两个测试用例捕获（ME 不满足） |

## 常见 MECE 错误

| 错误 | 表现 | 修正 |
|:---|:---|:---|
| 双重归因 | "客户不满意因为价格太高和服务太差" — 同一客户可落入两类 | 改为"价格型不满意 vs 服务型不满意 vs 两者兼有"（引入第三个类别） |
| 遗漏隐式类别 | "用户分 3 类：付费、免费试用、新注册" — 遗漏"已流失" | 补上第 4 类"已流失"，或收窄定义范围 |
| 分类粒度不统一 | "任务分：前端开发、数据库优化、修 Bug" — Bug 可能是前端的也可能是数据库的 | 先按模块分（前端/后端），再在模块内按类型分 |
| "其他"口袋类 | 大量项归入"其他" → CE 形式上满足但实质未穷举 | 进一步细分"其他"或重新定义分类体系 |
