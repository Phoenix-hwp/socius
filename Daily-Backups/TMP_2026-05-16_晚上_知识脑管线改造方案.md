# 知识脑管线改造方案（P0-P3 + 概念自述 + Step S）

> 来源：今晚消化《如何阅读一本书》后，对照书中分析阅读三阶段框架，对知识脑当前消化管线做了系统审查。
> 当前协议 19 份（CP-001~CP-019），概念树 Reading 域 10 个子概念。

---

## 一、问题发现（P0-P3 四盲区）

对照《如何阅读一本书》的分析阅读三阶段：

```
书中框架      知识脑当前管线        盲区
─────────    ─────────────        ────
检视阅读     Step 0 源分解         ✅ 已有
分类         分类器六类判别         ✅ 已有
诠释阶段     缺失                  P0 — 跳过了"用自己的话复述"的自检
评断阶段     缺失                  P1 — 没有内外部一致性验证
四问闸门     缺失（仅部分覆盖）    P2 — ①已覆盖，②③缺失，④已有activation
主题阅读     缺失                  P3 — 同一概念域积累多协议后的跨协议合成
```

### P0：缺少诠释自检（分析阅读第二阶段）

- **表现**：分类 → 提取骨架之间，跳过了"我到底读懂没有？"的验证
- **书中依据**：规则六——"检验是否理解的唯一标准就是能不能用自己的话重新表达"
- **当前提炼结果是"原文翻译成模板格式"，没有强制要求先自述**

### P1：缺少评断验证（分析阅读第三阶段）

- **表现**：知识源说什么就产什么协议，没有一致性检查
- **书中依据**：规则九～十一——批评式阅读的礼节
- **用户建议**：跟网络知识、大模型知识做一个简单比对互相验证

### P2：缺四问闸门汇总

- ① 整体在谈什么？ ✅ 分类器
- ② 细部说了什么？ ❌ 缺诠释自检
- ③ 有道理吗？ ❌ 缺评断验证
- ④ 跟我有什么关系？ ✅ activation 字段
- P2 本质是 P0+P1+分类+activation 的质量 gate

### P3：缺主题阅读

- 同一概念域下协议积累 ≥5 份后，触发跨协议合成
- 建议等全部知识卡片消化完后试做

## 二、P0-P2 改造方案

### P0：诠释自检（classifier.md 加 Step 1.5）

位置：`classifier.md` §三 分类流程，Step 1（读入）和 Step 2（判别主分类）之间。

内容：
- **一句话自述**：用自己的话说这段知识的核心主张
- **概念锚候选**：预填 concept_anchor（含领域+概念+简述，可用于回填 concept-tree.json 的 description 字段）
- **困惑标注**：有信心标注 confidence（high/medium），不阻塞流程

影响范围：仅 `classifier.md` 一个文件。

### P1：评断验证（extract-templates.md 加提炼后验证）

位置：`extract-templates.md` 提炼总则后，或作为独立步骤。

内容：
- **大模型自检**：新协议的概念锚是否与已有协议冲突
- **网络快扫**（WebSearch）：仅对公开知识源 + 核心概念可能有争议时触发
- **评断注释段**（写入协议底部）：一致性 / 外部验证 / 边界自知 / 来源质量

关键原则：**评断不阻断入库**，只标注。

### P2：四问闸门（汇总 gate）

位置：可用于消化完成时的 checklist，不增加新操作步骤。

内容：
- □ ① 整体在谈什么？— 分类已完成
- □ ② 细部说了什么？— 诠释自检通过
- □ ③ 说得有道理吗？— 评断验证已完成
- □ ④ 跟我有什么关系？— activation 字段已填写

## 三、概念自述（concept-tree.json 缺字段）

### 现状

- `concept-tree.json` 节点仅含：`parent / children / siblings / protocols / first_seen`
- `activation.md` 过滤器 1 的概念空间碰撞检测仅做**字符串前缀匹配**（`BPMN.activity` ∈ `BPMN.*`）
- 没有自然语言描述，激活系统无法做语义匹配

### 改造

`concept-tree.json` 每个节点加 `description` 字段：

```json
"Reading.Terminology": {
  "parent": "Reading",
  "children": [],
  "protocols": ["CP-014"],
  "description": "阅读方法中与作者达成共识的概念——找出关键术语并精确理解其含义",
  "first_seen": "2026-05-16"
}
```

激活匹配从纯前缀匹配升级为**语义优先**（前缀匹配兜底）。

### 与 P0 的衔接

P0 诠释自检中的"概念锚候选 + 简述"可直接回填为 `description` 字段。

## 四、Step S：收束自检（框架四个输出端中 ① 的自动化）

### 现状

框架 §二 定义四个输出端，其中 **① 审视现有系统** 目前仅手动触发。

### 建议

在消化管线末尾（协议落盘后）新增 Step S：

```
消化完成 → Step S：收束自检 → 闸门检查（P2）→ 标记完成
```

Step S 内容（轻量，不自动落盘改造系统的代码）：

1. **新知识与已有协议的关系**：互补 / 重叠 / 冲突？
2. **新知识是否暴露系统盲区**：分类器覆盖不到的知识类型？提炼模板需扩展？管线步骤缺失？
3. **若能优化，标记但不立即动手**：产出优化建议在对话中提出，下次讨论是否落地

## 五、改造总览

| 改造 | 目标文件 | 新增内容 | 难度 | 依赖 |
|:---|:---|:---|:---|:---|
| P0 诠释自检 | `classifier.md` | Step 1.5（三字段自述） | 低 | 无 |
| P1 评断验证 | `extract-templates.md` | 提炼后验证 + 协议评断注释段 | 中低 | 需定义网搜触发条件 |
| P2 四问闸门 | `classifier.md` 或 `extract-templates.md` | 消化完成 checklist | 极低 | 依赖 P0/P1 |
| 概念自述 | `concept-tree.json` + `concept-anchor.md` + `activation.md` | 加 description 字段 + 填写规范 + 语义匹配 | 中 | 需回填已有 19 份协议的概念锚 |
| Step S 收束自检 | `classifier.md` 或独立步骤 | 消化后的自我审查步骤 | 低 | 无 |

总计：改 3-4 个 Markdown 文件 + 1 个 JSON 文件，无脚本修改。
