## 变更影响防控方案（三层）

### 第一层：变更前影响枚举（最轻量，建议优先落地）

任何涉及路径/格式/字段名/文件名变更，**落盘前强制执行**：

```
rg -l "<变更关键词>" --glob "*.{py,mjs,md,mdc,json}"
```

将输出文件列表作为待修复清单，逐项打勾后执行。

> 写入位置建议：新建横切规则 `pre-change-impact-enumeration.mdc`，与 `pre-edit-script-change-brief.mdc` 同级。

### 第二层：影响面注册表

建 `change-impact-checklist.md`（与 `method-reliability-registry.md` 同级），按对象类型预注册搜索模式：

| 对象类型 | 搜索模式 | 涵盖范围 |
|:---|:---|:---|
| 脚本 | `rg "xxx" scripts/*.{py,mjs}` | 写入/读取/关联/入出 |
| Agent 规则 | `rg "xxx" .cursor/rules/*.mdc` | 框架/流程/路由 |
| Agent 模板 | `rg "xxx" Templates/*.md` | QnA 模板 |
| 数据文件 | `rg "xxx" --glob "*.{json,jsonl}"` | 索引/关联 |
| 文档 | `rg "xxx" --glob "*.md"` | 架构/README/Memo |

### 第三层：跨设备拉取后一致性检查

结合 `post-git-pull-three-layer-awareness.mdc`，拉取后自动检查 diff 涉及的模块是否有未同步引用。轻量版：跑 `rg "Knowledge_Cards"` 看残留引用。

# GitHub 技能获取方案（2026-05-11 晚上续）

## 背景

讨论为 Agent 接入 GitHub MCP Server（`@modelcontextprotocol/server-github`），从 GitHub 开源仓库中获取技能补充。

## 接入方案

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "<GitHub Token>"
      }
    }
  }
}
```

Token 仅需 `public_repo` 权限（读公开仓库）。

## 技能获取四步流程

| 步骤 | 名称 | 用户动作 | Agent 产出 |
|:---|:---|:---|:---|
| 1 | 语义展开 | 给粗略描述（如"审查 PR 的"） | GitHub 搜索候选仓库列表（星数/活跃度/简介） |
| 2 | 关键维度对齐 | 选 1-2 个最看重的维度 | 候选池 3-5 个，按维度打分对比表 |
| 3 | 一选一确认 | 选一个候选 | 接入预览：路径/入口文件/依赖/冲突检查 |
| 4 | 落地 + 登记 | 确认 | 落盘 + 更新技能清单 |

### 关键维度预设

| 维度 | 含义 |
|:---|:---|
| 适配 Cursor？ | 是否直接可挂载为 `skills-cursor/` |
| 依赖复杂度 | 零依赖 / npm 包 / 外部 API key |
| 覆盖度 | 单场景 vs 覆盖同类全场景 |
| 沟通负担 | 精简（适合对话）vs 长篇（适合独立阅读） |
| 维护活跃度 | 近 3 个月有更新 vs 已停更 1 年+ |

### 简化版

用户只给关键词 → Agent 直接出前三名（按"最适配 Cursor + 依赖最轻"排）→ 用户说"用第一个"或"换一批"。

## 两条底线

1. 搜索前不假设——先确认 GitHub 上实际有什么，再讨论
2. 对比必有量化——至少给星数、更新时间、依赖数三维度
