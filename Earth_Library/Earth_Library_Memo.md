#### 用户问题（原文要点）

关于地球图书馆，突然在想是不是做错了：更应该让 Agent **阅读 Notion 文章**，再**按一定规则在读取后重新归纳提炼**，并保存为**相关知识链路**；后续回答时再与**网上检索到的知识**配合，可能更符合实际任务场景、也更有效率。而不是像现在这样 **硬性采集** 的方式。

#### 助手回复（要点归纳）

1. **不一定是「做错」**，而是两种不同假设下的策略：**硬性采集**侧重可重复、批量、结构稳定、页级锚点与去重；**读后归纳**侧重任务导向、表达密度、语义边（链路）与回复可直接拼接。

2. **硬性采集**解决的主要是：不漏页、同一棵树多次跑结构一致、上下文不完全绑定单次对话。

3. **读后归纳**解决的主要是：同一材料在不同任务下的摘要角度、归纳时顺手写 prerequisite/对比/边界、库内段落更贴回答。

4. **代价对照**：归纳路径的批量成本更高、复现性弱于脚本；需注意幻觉与曲解（尤其未核对原文时）。

5. **常见折中**：**分层** — Notion/原文作档案层（求全）；Earth Library 卡片作工作层（求用）；**骨架可采集，血肉靠读后归纳**。

6. **效率**：单次对话里归纳往往更贴任务；跨会话若无固定流水线易口径漂移 — 硬性采集保证锚点一致。

7. **后续可选**：不推翻现有卡片，另增「读后归纳」工作流与模板（篇幅、必须 SourceURL/页码、禁止无引用事实断言等）；若彻底以 Agent 归纳为主，需固定何时读 Notion、模板、谁复核、如何更新旧卡。
