# 一句话定义
UML 状态图表达"一个对象"的状态迁移过程，通过状态（圆角方框）→ 转移（带操作名的箭头）→ 内部转移（回环）三要素描述状态变化，与流程图互补——流程图表达活动迁移，状态图表达状态迁移。

# 核心内容

## 状态图基本元素
| 元素 | 符号 | 含义 |
|:---|:---|:---|
| 状态 | 略方的圆角矩形 | 对象所处的状态（如"待审核""已通过"）|
| 转移 | 带箭头直线 + 操作名 | 从一个状态转移到另一个状态 |
| 内部转移 | 带箭头回环 | 操作不改变状态 |
| 开始/结束 | 实心圆 / 双圆 | 与流程图相同 |

## 绘制四步法
1. **绘制主干的状态**：先忽略次要分支，画出核心主线
2. **进行状态的拆合**：用两个词表述状态后确认是否要拆分（如"已提交，待审核"）
3. **完善分支的状态**：寻找与主干状态相反的状态作为分支
4. **完善角色和操作**：思考每个角色能否将当前状态转移到任意状态

## 状态图 vs 流程图
| 维度 | 状态图 | 流程图 |
|:---|:---|:---|
| 表达什么 | 对象的状态 | 活动/行为 |
| 线上写什么 | 操作/活动名称 | 判断条件（通常不写活动名）|
| 何时用 | 一个对象有多种状态需管理 | 包含若干步骤的流程且判断条件多 |
| 不可有菱形 | 是（更简洁和无歧义）| 否（菱形是核心）|

## 关键设计原则
- **有对象才有状态**：一个状态图只表达"一个对象"的状态，建议在图中写出对象名称
- **"编辑中"状态无需画入状态图**：它与草稿/待审核/审核通过不在同一层次
- **锁的机制**：用户进入编辑界面时加锁，避免两个人同时编辑导致数据覆盖

# 使用场景
- 订单流程、商品审核、优惠券使用等有多状态的对象必须画状态图
- 登录流程不需要状态图（没有值得管理的状态）
- 梳理完后作为开发和测试的"状态真值表"

# 与其他框架的关系
- 与 **UML 流程图**：先画流程图再画状态图；状态图中转移线上的操作名恰对应流程图的活动名
- 与 **UML 类图**：类图定义对象的数据结构，状态图描述对象在其生命周期中的状态变化
- 与 **BPMN**：BPMN 侧重业务流程，UML 状态图侧重对象状态管理——层面不同但互补

# 一条边界
一个状态图只能表达"一个对象"的状态。如果发现自己在同一个图中描述了多个对象的状态变化，应立即拆分。判断标准：状态名称如果主语不一致（如"用户已提交"和"订单已发货"），说明混入了两个对象。
