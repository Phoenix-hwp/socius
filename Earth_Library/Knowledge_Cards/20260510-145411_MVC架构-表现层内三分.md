# Summary
MVC架构：表现层内三分

# Details
# 核心内容

## 定义

MVC（Model-View-Controller）是一种**表现层内部的架构模式**，将用户界面相关职责切分为三个角色：

| 角色 | 职责 | 典型形式 |
|:---|:---|:---|
| Model | 持有视图状态，响应查询与更新 | 前端状态对象 |
| View | 渲染 UI，展示数据 | HTML / 组件模板 |
| Controller | 接收用户输入，编排 Model 与 View 的交互 | 事件处理函数 / 路由 |

## 在系统分层中的位置

MVC 运作在**三层架构的表示层（Presentation）内部**，不直接涉及业务逻辑层和数据访问层。

```
表示层（Presentation Layer）
├── Model（视图状态）
├── View（UI 渲染）
└── Controller（用户输入处理）
     ↓ HTTP / API 调用
业务逻辑层（Business Logic）
     ↓
数据访问层（Data Access）
```

关键区分：MVC 的 Model ≠ 业务层的 Domain Model。MVC 的 Model 是前端持有的视图状态（View State），而非领域规则。

## 常见误用

| 误用 | 实际情况 |
|:---|:---|
| "MVC 就是三层架构" | MVC 只在表示层内，三层是系统级纵切 |
| "后端 Controller→Service→DAO 就是 MVC" | 那是三层架构的变体，M（Model）裂为 Service 和 DAO 两层 |
| "MVC 过时了" | 其思想被 MVVM、MVP、Flux/Redux 继承和发展 |

## 历史演进

```
MVC（1979，Smalltalk）
  → MVP（1990s，解耦 View 与 Model）
    → MVVM（2005，数据绑定）
      → Flux / Redux（2014，单向数据流）
```

MVVM 是目前前端框架（Vue、React 配合状态管理）最常见的 MVC 后继模式。

## 与三层架构的关系

MVC 是最多人的架构入门，三层架构是 MVC 的系统级泛化——把 Model 从"前端状态"升级为"整个系统的数据与规则"。整洁架构和 DDD 则在此基础上进一步引入依赖倒置和领域建模。

## 与规则体系三层的距离

MVC → 三层架构 → 整洁架构/六边形 → DDD → 规则体系三层，这是一条关注点分离不断深化的演进线，但每一代解决的核心问题不同。
