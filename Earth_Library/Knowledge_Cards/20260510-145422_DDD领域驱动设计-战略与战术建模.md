# Summary
DDD领域驱动设计：战略与战术建模

# Details
# 核心内容

## 定义

领域驱动设计（Domain-Driven Design，Eric Evans 2003）不是一种层结构，而是一整套围绕**领域建模**展开的思想体系。核心信念：软件的核心复杂度在于领域本身，不在技术。

## 两大板块

```
DDD
├── 战略设计（Strategic Design）
│   ├── 限界上下文（Bounded Context）
│   ├── 通用语言（Ubiquitous Language）
│   ├── 上下文映射（Context Map）
│   └── 子域划分（核心域 / 支撑域 / 通用域）
│
└── 战术设计（Tactical Design）
    ├── 分层架构
    ├── 实体（Entity）、值对象（Value Object）
    ├── 聚合（Aggregate）、聚合根（Aggregate Root）
    ├── 仓储（Repository）、工厂（Factory）
    ├── 领域服务（Domain Service）、领域事件（Domain Event）
    └── 应用服务（Application Service）
```

## 战术分层（标准四层）

```
Interface（接口层） → Application（应用层，极薄编排）
     → Domain（领域层，核心业务规则，不依赖任何外层）
         → Infrastructure（基础设施层，实现领域层定义的接口）
依赖方向：外层 → 内层（与整洁架构一致）
```

| 层 | 职责 | 关键约束 |
|:---|:---|:---|
| Interface | 接收请求，转换格式 | 不含业务逻辑 |
| Application | 编排用例，事务管理 | 极薄，无业务规则 |
| Domain | Entity、ValueObject、Aggregate、DomainEvent | 不依赖任何外层 |
| Infrastructure | ORM、消息队列、外部 API 实现 | 实现 Domain 定义的接口 |

## 聚合（Aggregate）— DDD 最核心的概念

一组强一致性的领域对象，外部只能通过聚合根访问：

```
Order（聚合根）
├── orderId: OrderId（值对象）
├── status: OrderStatus
├── items: List&lt;OrderItem&gt;（实体）
└── totalAmount: Money（值对象）

规则：
1. 外部只能通过 Order.addItem() / Order.confirm() 修改
2. 聚合内保证事务一致性
3. 不同聚合之间仅通过 ID 引用
4. 聚合大小以"一次事务能改完"为上限
```

**聚合的难点在"切多小"**：太大则锁竞争严重，太小则业务不变量无法保证。

## 战略设计核心概念

| 概念 | 核心问题 |
|:---|:---|
| 限界上下文 | "这个词在不同模块中含义一样吗？"——是微服务边界的理论依据 |
| 通用语言 | "开发和业务说的是同一套术语吗？"——代码类名应直接反映业务术语 |
| 上下文映射 | "两个上下文如何协作？"——共享内核、客户-供应商、防腐层（ACL） |
| 核心域/支撑域/通用域 | 有限资源聚焦核心域，通用域可外购 |

## 与其他架构的关系

```
MVC（表现层内三分）
  → 三层架构（系统级纵切）
    → 整洁架构（依赖倒置约束）
      → DDD（填充领域建模工具）
        → 微服务（以限界上下文为服务边界）
```

DDD 的战略设计回答"服务边界怎么划"，战术设计回答"边界内部怎么建"。
