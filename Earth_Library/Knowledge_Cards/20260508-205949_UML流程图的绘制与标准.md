# 一句话定义
UML 流程图分三层（业务→交互→实现），从整体到细节、从设计到实现递进，核心原则是"手递手的交互"（人-人 / 人-机各层均保证一个活动后紧跟下一个参与者的一个活动）。

# 核心内容

## 三层次流程图
| 层次 | 表达对象 | 目标 | 核心原则 |
|:---|:---|:---|:---|
| 业务流程图 | 人与人的交互 | 厘清和设计业务 | 手递手人人交互、去掉页面交互、去掉系统实现 |
| 交互流程图 | 人与机器的交互 | 指导原型图绘制 | 涉及规则的步骤要画出来、手递手人机交互 |
| 实现流程图 | 机器在做什么 | 设计软件 | 由研发人员绘制 |

## 绘制三步法
1. **画主流程**：先粗后细，加入分支
2. **完善细节**：加入异常，拆出流程
3. **加入泳道**

## 异常流程四类
| 类型 | 举例 | 处理 |
|:---|:---|:---|
| 规则限制 | 库存不足、权限不够 | 明确限制条件与提示 |
| 用户不操作/超时 | 不操作、操作慢 | 超时倒计时、自动取消 |
| 操作错误 | 输入错误、反复输错、反复点击 | 防重复提交、锁定账户 |
| 撤回操作 | 过程中反悔、完成后反悔 | 提供"上一步"按钮、事后修改/撤回 |

## 标准符号
- **活动**：圆角矩形 + 名称
- **开始/结束**：实心圆 / 双圆
- **判断/分支**：菱形（建议写判断条件在菱形内 + 线旁用方括号标注条件）
- **选择**：直接从活动延伸两条线并标注选择内容
- **合并**：多线进一线出（可用菱形或直接汇总）
- **并行/汇合**：粗横线

# 使用场景
- 梳理一个新业务的完整流程图（先业务流程，再交互流程）
- 页面交互较复杂时画交互流程图指导原型图
- 团队评审流程时检查异常分支是否完备

# 与其他框架的关系
- 与 **BPMN**：BPMN 是业务流程建模的国际化标准符号体系，UML 流程图更适合产品层面的交互表达
- 与 **UML 状态图**：先画流程图再画状态图；流程图表达活动之间的迁移，状态图表达状态之间的迁移
- 与 **UML 类图**：流程图描述流程，类图描述数据结构——两者互补

# 一条边界
业务流程图的三个原则（手递手人人交互、去掉页面交互、去掉系统实现）是"默认真理"，但在页面交互不完成业务就进行不下去时，必须加入页面交互——原则二存在让步条件。不要教条式地完全不画页面交互。
