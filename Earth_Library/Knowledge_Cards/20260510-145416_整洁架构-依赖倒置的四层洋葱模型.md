# Summary
整洁架构：依赖倒置的四层洋葱模型

# Details
# 核心内容

## 定义

整洁架构（Clean Architecture，Robert C. Martin 提出）是一种以**依赖倒置**为核心的架构风格。最内层是最高抽象，外层依赖内层，不可逆。

## 四层洋葱模型

```
外层
├── Frameworks & Drivers（框架与驱动层）
│      Web 框架、数据库驱动、UI、外部设备
│   ┌── Interface Adapters（接口适配层）
│   │      Controller、Presenter、Gateway
│   │   ┌── Use Cases（用例层 / 应用业务规则层）
│   │   │      具体业务规则、应用逻辑
│   │   │   ┌── Entities（实体层 / 企业业务规则层）
│   │   │   │      核心领域模型，最抽象的规则
│   │   │   └────────────────
│   │   └────────────────
│   └────────────────
└────────────────
依赖方向：外层 → 内层（单向，不可逆）
```

## 核心原则

1. **依赖只能从外向内**：内层不知道外层的存在
2. **Entity 不知道有人在用它下单**——只定义"订单是什么"
3. **Use Case 不知道请求来自 HTTP 还是 CLI**——只管业务编排
4. **Adapter 知道两边**——负责格式翻译（JSON → UseCaseInput）
5. **Framework 是插件**——可以被替换而不影响业务

## 与 DDD 的关系

整洁架构是房子的结构规范，DDD 是房间里每个房间的布置原则。DDD 的 Domain 层和 Application 层可以"嵌"在整洁架构的 Entities + Use Cases 层内。两者常组合使用。

## 与三层架构的对比

| 维度 | 经典三层 | 整洁架构 |
|:---|:---|:---|
| 驱动原则 | 按类型分层（UI/业务/数据） | 按稳定性分层（稳定→多变） |
| 依赖方向 | 上层调下层（宽松） | 外层依赖内层（严格不可逆） |
| 核心价值 | 简单、入门低 | 长期演进可维护 |

## 关键反模式

- 内层引用外层的类（如 Entity 依赖 ORM 框架）
- Use Case 包含 SQL 或 HTTP 细节
- 跨层传框架专有对象（如直接把 HttpRequest 传到 Use Case 层）

## 判断问题

| 问题 | 答案 |
|:---|:---|
| "这个改动会让内层模块重新编译吗？" | 如果是 → 违反了依赖方向 |
| "这个模块换个数据库需要改吗？" | 如果是 → 基础设施没有解耦 |
