# 一句话定义
通过七个维度（What/Why/Who/When/Where/How/HowMuch）系统化拆解问题，每个维度从用户侧和设计侧双向交叉审视，避免遗漏关键信息。

# 核心内容

| 维度 | 用户侧 | 设计侧 |
|:---|:---|:---|
| What（是什么） | 功能提供什么服务？与用户需求是否一致？ | 背景、优先级、具体交付物 |
| Why（为什么） | 用户的目标是什么？要解决什么痛点？ | 为什么做这件事？用户价值与商业价值平衡点？ |
| Who（谁） | 目标用户是谁？角色间如何交互？ | 谁做？是否需要调配资源？ |
| When（什么时候） | 什么时间使用？不同时段需要不同服务吗？ | 上线时间？迭代节奏？ |
| Where（哪里） | 使用场景是什么？场景存在哪些问题？ | 存在于哪些功能上？多端兼容？ |
| How（具体如何） | 用户如何通过产品完成任务？ | 如何开展设计？为何这样设计？ |
| How Much（所需成本） | 使用频率？定价策略？ | 估算成本、风险及是否需取舍 |

每个母问题可延伸出子问题以深究分析对象，是设计方法论中复用性最高的分析工具。

# 使用场景
- 接到一个模糊需求时，用 7 个维度快速穷举「该问什么」
- 需求评审前，按 7 维度自检是否有遗漏视角
- 跨职能沟通（运营/研发/设计）时，作为对齐口径的清单

# 与其他框架的关系
- 与**需求分析十三要素五步法**：5W2H 是拆解问题的通用框架；十三要素是需求分析领域的专项深化，补全了「角色区分」「频率」「横向替代/纵向互补」等维度
- 与**SWOT 分析法**：互补 — 5W2H 拆解「怎么做」，SWOT 评估「做这件事的内外条件」
- 与**产品起步逻辑**：5W2H 中的 Who/When/Where 可为「发现」模块的渠道和场景分析提供输入

# 一条边界
5W2H 是发散型清单工具，不提供收敛机制（不帮你排序或取舍）。适合前期穷举，不适合需要收敛决策的场合，需搭配 KANO 或优先级排序框架使用。
