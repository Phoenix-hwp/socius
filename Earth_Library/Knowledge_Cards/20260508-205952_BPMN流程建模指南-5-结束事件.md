# 一句话定义
BPMN 结束事件表示流程的终点，有通用/消息/错误/升级/取消/补偿/信号/终止/多重 9 种类型，各自定义不同的结束结果（消息通知/错误终止/补偿回滚/信号广播等）。

# 核心内容

## 九种结束事件速查
| 类型 | 结果 | 对其他活动线程的影响 |
|:---|:---|:---|
| 通用 | 无预定义结果 | 各路独立完成 |
| 消息 | 结束时向参与者发送消息 | 各路独立 |
| 错误 | 生成错误，终止所有当前活动线程 | **全部终止** |
| 升级 | 触发升级 | 其他线程不受影响，继续执行 |
| 取消 | 取消事务并触发边界取消中间事件 | 用于事务子流程 |
| 补偿 | 触发已完成活动的补偿 | 按顺序流相反顺序补偿 |
| 信号 | 广播信号到所有能接收的进程 | 跨进程/跨池广播 |
| 终止 | 立即结束所有活动（含多实例）| **全部终止，无补偿或事件处理** |
| 多重 | 多种结果全部发生（如发送多消息）| 各路独立 |

## 关键区分
- **错误 vs 终止**：错误=可被边界错误事件捕获并进入异常流；终止=一切立刻结束，不可干预
- **普通结束 vs 终止**：普通结束=各路独立完成不影响彼此；终止=一路结束全停
- **消息 vs 信号**：消息有指定目标和来源；信号是广播，无特定目标

# 使用场景
- 流程有多种结束分支时，选择合适的结束事件区分不同结果
- 事务流程中使用取消和补偿实现回滚逻辑
- 跨系统通知场景使用消息或信号结束事件

# 与其他框架的关系
- 与 **BPMN 指南(4)开始事件、(6)中间事件**：事件三章各自覆盖一个阶段
- 与 **BPMN 指南(8)事件常见错误**：错误 4 专门纠正"混淆结束/终止事件"

# 一条边界
结束事件只能"抛出"结果，不能"捕获"触发器。如果流程结束时需要等待外部确认再终止，应使用"中间消息捕获事件"而非结束事件。不要在普通流程中滥用终止事件。
