# Summary
UML 绘制方式与通用注意点

# Details
## 一、绘制方式（怎么做）

### 1. 先定目的再选图

按需选用 UML 图种，避免「全套画满」：

| 目的 | 常用图 |
|:---|:---|
| 业务流程 / 用例边界 | 用例图 |
| 对象结构与关系 | 类图 |
| 调用顺序、异步与消息 | 序列图 |
| 状态与生命周期 | 状态机图 |
| 活动、分支与合并 | 活动图 |
| 部署与节点 | 部署图 |
| 组件边界与依赖 | 组件图 |

### 2. 粒度分层

- **概念层**：少属性，强调角色与关系；面向产品或架构对齐。
- **设计层**：类名、关键操作、关联多重性；面向开发与实现对照。
- **实现层**：可与代码一一对应，适合逆向工程或局部精读；不必全系统维持该粒度。

### 3. 工具与载体

- **建模工具**：Enterprise Architect、StarUML、PlantUML、Draw.io（版型与连线）、Mermaid（文档内嵌）。
- **白板优先**：先手绘或随意框图定结构，再迁入工具规范化，通常比直接抠符号更高效。

### 4. 协作习惯

- 采用「主图 + 若干局部放大子图」，优于单张巨型图。
- 图旁用一两句话说明「本图要回答什么问题」，降低误读。

## 二、注意点（易错与维护）

### 1. 语义与符号

- **关联 / 依赖 / 组合与聚合**：关联表示相对稳定结构关系；依赖表示「用过即散」的引用；组合与聚合涉及所有权与生命周期，标错会误导设计。
- **箭头方向**：泛化（继承）、接口实现、依赖、有向关联方向不一；团队内固定约定并在图例中说明。
- **序列图**：同步调用与返回（实线调用、虚线回复）需区分；异步、回调要用对应消息类型表达，否则看不出并发语义。

### 2. 抽象级别

- 同一图中避免混用「领域概念」与「具体框架类名」，除非标明为实现视图。
- 类图不必罗列全部 getter/setter；突出不变式与关键协作接口即可。

### 3. 完整性与维护成本

- UML 与代码长期双向同步成本高；通常仅对**稳定边界**（对外 API、核心领域模型、关键流程）保持图档更新。
- 迭代时在注释或变更说明中写清「相对上一版的变化」，常比重画整张图更有用。

### 4. 可读性

- 控制元素数量：单图建议不超过约 15～20 个主要节点，超出则分包或分层。
- **命名**与代码、词汇表一致（中英统一），避免同一概念多名并存。
- 若用颜色编码（如聚合根、外部系统），必须在图例中说明，避免无色或打印场景失效。

### 5. 活动图与流程图

- 活动图适合泳道（角色）、分叉汇合、对象流；若仅为线性步骤，简单流程图有时更清晰，不必强行使用 UML 活动图。
