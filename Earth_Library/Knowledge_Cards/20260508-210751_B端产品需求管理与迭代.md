# 一句话定义
B端产品需求的三层分类（业务/用户/技术）与四类价值（商业/业务/用户/技术）的映射框架——理解需求的层次是需求管理和优先级排序的基础。

# 核心内容

## 原始需求 vs 产品需求

| | 原始需求 | 产品需求 |
|:---|:---|:---|
| 来源 | 用户/产品经理采集的未分析需求 | 分析挖掘后的解决方案 |
| 确定性 | 充满不确定性（痛点 or 方案） | 颗粒度清晰、逻辑严谨 |
| 关系 | 多对多（可拆分为多产品需求/与其他原始需求合并） | 对应多个研发/测试任务 |

## 需求的三层分类

| 层次 | 来源 | 分析方式 | 承载价值 |
|:---|:---|:---|:---|
| **业务需求** | 中高层（自顶向下）+监管/政策 | 业务诊断+DDD建模+BPR流程再造 | 业务价值（帮业务单元达成业务目标） |
| **用户需求** | 一线用户+基层管理者（自底向上） | 可用C端方法论：客户旅程地图/KANO模型 | 用户价值（满意度+操作效率） |
| **技术需求** | 系统架构与功能完备性 | 功能复用性+系统间架构合理性 | 技术价值（架构合理+避免重复开发） |

## 四类价值（最终都服务于商业价值）
商业价值 → 业务价值 + 用户价值 + 技术价值（三者→商业价值）

**注**：马斯洛模型不完全适用于B端需求管理——B端面向组织/业务/复杂系统，需从多维审视需求类型和层次。

## 需求提报规范
甲方公司可通过需求提报模板提升需求质量（需填写需求价值+预期收益+度量方式）。乙方不宜强制模板。

# 使用场景
- 收到模糊的「原始需求」时，用它拆解为业务/用户/技术三层，判断真实归属
- 需求评审前，用三层+四价值框架评估每个需求的实际价值和紧迫度
- 需求池长期混乱时，用此框架重新分类→找出未被覆盖的维度（如只有业务需求无用户需求）

# 与其他框架的关系
- 与**KANO 模型**：B端用户需求层可用 KANO 做功能优先级排序（基础/亮点/期望/无差别/反向）
- 与**需求分析十三要素五步法**：本卡关注需求分类与价值判断；十三要素关注单个需求的深度分析——可串联
- 与**B端市场分析与业务调研**：调研输出的「业务分析三层框架」（战略→战术→执行）与本卡的需求三层可在产品设计中串联

# 一条边界
三层分类是分析框架，不是所有需求都要分为三层才处理——简单需求可直接进开发。另，「技术需求」容易被滥用为「重构需求」的遮羞布；判断是否为纯技术需求的关键是：是否有可验证的技术价值（降低维护成本/提升复用性/改善架构合理性）。
