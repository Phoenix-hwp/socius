# 一句话
覆盖"怎么理解用户、怎么画用户旅程、怎么站在用户视角发现问题"，是产品方法论域的天然前置——先用这里的方法理解用户，再用产品方法论做决策。

# 领域地图

## 场景 → 工具速查
| 你想做什么 | 用哪张卡 | 一句话说明 |
|---|---|---|
| 可视化用户使用产品的完整体验 | 用户体验旅程图 | 从用户视角出发，以叙事方式展示行为→情绪→痛点→机会点，是发现设计机会点的核心工具 |
| 动手画一张用户旅程图 | 如何绘制「用户旅程图」| 九步标准流程 + 七要素，从用户画像到机会点完整操作指南，含常见误区 |
| 站用户视角发现痛点（不画操作路径） | 同理心地图 | 六象限头脑风暴工具（说/想/做/听/看/痛点和收益），设计前后均可使用 |

## 常见组合路径
- **路径1**：理解一个新用户群体 → 先用同理心地图做头脑风暴（站用户视角，不拘泥于操作路径）→ 再用如何绘制用户旅程图画出完整路径 → 最后用用户体验旅程图做可视化呈现和机会点标注
- **路径2**：分析现有产品的体验问题 → 先用用户体验旅程图还原用户全流程 → 标注情绪曲线和痛点 → 用同理心地图深挖某一环节的底层动机
- **路径3**：设计评审的体验输入 → 先跑用户旅程图找到关键挫折点 → 交叉到产品方法论域的 KANO 模型做功能优先级判断 → 交叉到需求分析做方案设计

## 域内缺口
- **不含用户访谈方法论**：不覆盖深度访谈提纲设计、田野调查、日记法等
- **不含定量用户研究方法**：不覆盖问卷调查设计、数据分析、NPS 指标计算等
- **不含人物画像（Persona）创建**：同理心地图和旅程图都用到画像，但"如何创建画像"本身不覆盖
- **不含可用性测试**：不覆盖可用性测试的脚本设计、录制、SUS 量表等
- **Agent 遇到上述缺口问题时，优先走网络搜索**

## 交叉域参考
- 涉及"从用户洞察到功能设计"时，先用本域理解用户，再用《产品方法论》域概览卡（需求分析 / KANO / 5W2H）做功能决策
- 涉及"B 端用户的特殊调研方法"时，参考《B 端产品》域概览卡（市场分析与业务调研 / 多利益方识别）
- 涉及"用户旅程对应业务流程"时，参考《流程建模与架构》域概览卡，将用户旅程中的触点映射到 BPMN 流程节点
