# 一句话定义
将产品功能按「实现程度」与「用户满意度」的关系分为五类（基础/亮点/期望/无差别/反向），帮助产品经理在有限资源下做功能取舍排序。

# 核心内容

| 类型 | 特征 | 满意度曲线 | 应对策略 |
|:---|:---|:---|:---|
| **基础功能**（必备属性） | 没有→极其不满；有了→理所应当 | 趋近 0 但永不破 0 | 留足资源，必须做 |
| **亮点功能**（魅力属性） | 没有→无所谓；有了→惊喜 | 从 0 正相关增长 | 优先成本低的；靠洞察用户心智 |
| **期望功能**（期望属性） | 越多越满意，越少越不满 | 直线正相关 | 选性价比高的 |
| **无差别功能** | 做与不做，用户无感 | 接近水平线 | 低成本原型验证后决定 |
| **反向功能** | 做得越多用户越讨厌 | 负相关下穿 0 | 权衡多方利益，不可简单放弃 |

**功能演变规律**：随时间推移，亮点功能→期望功能→基础功能，行业门槛随之提升。

**KANO 调查表**：正向（有了感觉如何）+ 反向（没有感觉如何）双问，对照 5×5 矩阵判定属性类别。

# 使用场景
- 版本迭代规划时，对候选功能清单做 KANO 分类，区分「必须做」和「锦上添花」
- 接到大量用户反馈但资源有限时，通过 KANO 调查表量化判断优先级
- 竞品功能对标后，判断哪些是基础功能（不做不行）、哪些是亮点功能（可后发制人）

# 与其他框架的关系
- 与**5W2H 分析法**：5W2H 发散穷举功能维度，KANO 收敛排序 — 常见串联使用
- 与**产品各阶段关键指标**：KANO 分类随产品生命周期变化而演变，需结合阶段判断
- 与**点子过滤器**：互补 — KANO 解决「功能优先级」，点子过滤器解决「创意筛选」
- 与**B端产品需求管理**：B端场景中 KANO 需结合业务需求/用户需求/技术需求三层分类使用

# 一条边界
KANO 模型依赖用户调研数据，但用户说不出的「亮点功能」无法通过问卷发现 — 亮点靠产品人对用户心智的洞察，不在 KANO 统计范围内。此外，分类结果随时间推移会变化，不可一次调查一劳永逸。
