# 一句话定义
从八个维度（企业/用户/产品/交互/视觉/开发/运营/社会责任）对产品设计做结构化审视，避免仅凭审美偏好做主观评价。

# 核心内容

## 八维度评价清单

| 维度 | 核心问题 | 关键判据 |
|:---|:---|:---|
| **企业侧** | 设计是否帮助达成商业目标？ | 看设计是否挖掘出新机会点；脱离商业/业务背景的讨论无意义 |
| **用户侧** | 用户是否受益？付出多少代价？ | ① 是否解决本质问题；② 用户付出代价（学习/操作/迁移成本）是否划算；③ 是否有附加价值超出预期 |
| **产品侧** | 功能是否覆盖所有必要场景？ | 场景覆盖越全面，设计完整性越高；从「满足需求」升华到「提供更高效服务」 |
| **交互侧** | 是否做到自然、给用户自由？ | 防错机制、及时反馈、交互一致性、场景化设计、情感化设计、简单清晰的任务路径、用户心智模型；本质是降低人机交流摩擦度 |
| **视觉侧** | 信息是否高效传达？ | 界面信噪比是核心但非唯一标准；B端信噪比高≠设计差，关键是用户能否在臃肿界面中快速获取有用信息 |
| **开发侧** | 性能是否支持设计落地？ | 兼容性、运行效率、渲染性能；低配设备需降级处理（动效/动画降级为静态） |
| **运营侧** | 用户从哪里开始体验？第一印象是什么？ | 运营手段支撑产品正常运行：用户触达渠道、内容触动方式、第一印象设计 |
| **社会责任** | 产品是否有人情味？ | 保障用户资金利益、减少使用风险；为视障/语言障碍/极端情绪者提供人文关怀 |

## 关键洞见
- **交互本质**：做到自然，给用户自由，尊重用户的使用习惯
- **B端 vs C端子维度差异**：B端信息密度天然高，不能用 C端的信噪比标准评价
- **附加价值公式**：用户回报 > 付出 → 设计优秀；回报 = 付出 → 理所当然；回报 < 付出 → 不可取

# 使用场景
- 设计评审时，用八维度清单替代「我觉得不好看」的主观评价
- 竞品体验分析时，按八维度逐项打分输出结构化对比
- 设计团队能力建设时，按维度识别团队短板

# 与其他框架的关系
- 与**好产品的评价标准**：本卡聚焦设计层面，是「好产品评价标准」中「好用」层面的专项展开
- 与**KANO 模型**：企业侧的「商业价值」判断可借助 KANO 区分基础/亮点/期望功能
- 与**UML 类图**：产品侧「场景覆盖」分析可借助 UML 类图做系统化信息建模

# 一条边界
八维度是评价框架而非设计方法 — 你无法用本卡直接产出设计方案。它适合「评审已有设计」，不适合「从零生成设计」。另外，交互/视觉侧大量依赖设计原则积累，本卡只提供检查维度不替代具体设计知识。
