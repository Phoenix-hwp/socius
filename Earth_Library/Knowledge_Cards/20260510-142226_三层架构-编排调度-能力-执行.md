# Summary
三层架构：编排调度—能力—执行

# Details
# 核心内容

## 定义

三层架构是一种**关注点分离**的设计模式，将复杂系统按职责纵向切分为三层：

| 层 | 核心职责 | 比喻 |
|:---|:---|:---|
| 编排调度层 | 接收任务→查标签→路由到执行层→异常兜底 | 总调度台 |
| 能力层 | 定义契约（"我能做什么""调用我需要什么"） | 贴了标签的盒子 |
| 执行层 | 逐步执行具体操作，可含子工作流管道 | 盒子里的人 |

## 详细机制

### 编排层（Orchestration）
- 统一接收任务入口
- 读取能力层的"标签"（操作分型表）做路由
- 不关心执行细节，只关心"谁来做"
- 提供全局异常兜底

### 能力层（Capability）
- 本质：**贴了标签的盒子**——标签正面写"我能做什么"，标签背面写"调用契约"
- **不参与运行时调用链**——编排层直接路由到执行层，能力层是被读的地图而非中转站
- 一个能力可以有多个实现（如 Python+Node 双运行时）
- 一个脚本可以实现多个能力
- 盒子内归拢所有相关执行层工作流和脚本

### 执行层（Execution）
- 逐步执行具体操作
- 内部可以有**子工作流管道**（嵌套编排）
- 执行过程中对照能力层约束做自检
- 可由多脚本协同（subprocess CLI 调用）

## 横切关注点

不归入任何纵层的全局约束（如密钥保护、编码规范、跨设备路径）——像建筑规范，每层都要遵守但不属于某一层。

## 关键反模式

| 反模式 | 说明 |
|:---|:---|
| 编排层泄漏执行细节 | 编排层直接拼 API URL |
| 执行层做路由决策 | 工作流里判断"该走哪个模块" |
| 能力层写死具体步骤 | 框架里写"第1步：curl..."而非指针 |

## 判断框架

| 问题 | 答案指向 |
|:---|:---|
| "改调用顺序还是改具体操作？" | 顺序→编排层，操作→执行层 |
| "改动会影响其他模块的编排吗？" | 是→能力层（改契约），否→执行层 |
| "这个规则适用于所有模块吗？" | 是→横切规则，否→归属对应模块的三层 |
