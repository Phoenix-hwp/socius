# 一句话定义
BPMN 活动是业务流程中执行的工作单元，包含任务（原子）、子流程（复合可展开）和编排任务（跨参与者消息交互）三层次，并有服务/用户/手动/业务规则/脚本/发送/接收等多种任务类型标识。

# 核心内容

## 活动三层次
| 层次 | 特征 | 典型符号 |
|:---|:---|:---|
| 任务（Task）| 原子活动，无法进一步细分 | 圆角矩形 |
| 子流程（Sub-Process）| 可展开为内部流程 | 圆角矩形 + 展开标记（折叠/展开两种状态）|
| 编排任务（Choreography Task）| 涉及两个参与者的消息交换 | 上下带区显示参与者名称 |

## 常用任务类型速查
| 类型 | 何时使用 | 标志 |
|:---|:---|:---|
| 通用任务 | 工作单元无需再细化 | 无特殊标记 |
| 服务任务 | 由 Web 服务或自动化应用执行 | 齿轮图标 |
| 用户任务 | 人在软件辅助下执行（工作流型）| 人物图标 |
| 手动任务 | 无引擎/应用辅助执行 | 手图标 |
| 业务规则任务 | 向规则引擎输入并获得计算输出 | 表格图标 |
| 脚本任务 | 由流程引擎执行脚本 | 脚本图标 |
| 发送任务 | 向外部参与者发送消息即完成 | 深色信封 |
| 接收任务 | 等待外部参与者消息到达即完成 | 空心信封 |

## 活动属性
活动可叠加多种属性标记：循环（重复执行）、多实例（不同数据集多次执行）、补偿（仅在取消交易时执行）。组合后如"基于服务的可重复补偿任务"。

# 使用场景
- 选择合适的任务类型以准确表达流程中的自动化/人工边界
- 判断一个步骤应该用任务还是子流程（可分解性 vs 复杂度）
- 与外部系统对接时选择发送/接收任务表达消息交互

# 一条边界
区分任务和子流程的标准是"能否进一步分解"而非"复杂度"。简单活动也可建模为子流程如果建模者决定对其展开。不要误以为复杂=子流程、简单=任务。
