# Summary
构建自学习体系：方法可靠性注册表（失败自动降权）+ 能力模式识别与固化（重复任务序列自动封装）。

# Details

## 背景与痛点

在日常执行中发现两个痛点：
1. **同一任务类型**，某些执行方法屡次失败但下一轮仍被优先尝试（如 Shell 执行 git 多次失败但未记录）
2. **频繁出现的子任务组合**没有固化，每次都需要重新推理步骤

## 组件 A：方法可靠性注册表（Method Reliability Registry）

### 核心数据结构

| 任务类型 | 方法路径 | 总调用 | 连续失败 | 最后状态 | 最近上下文 | 状态 |
|---------|---------|--------|---------|---------|-----------|------|
| 提交git | Shell(git) | 12 | 3 | 失败(认证) | 2026-05-08 Win10 | ⚠ 降权 |
| 提交git | Shell(gh) | 5 | 0 | 成功 | 2026-05-10 Win10 | ✅ 正常 |
| Notion创建 | MCP | 20 | 2 | 失败(超时) | 2026-05-09 Win10 | ⚠ 降权 |

### 学习规则（阈值可配置）

- 连续失败 ≥ **3 次** → 标记为 `⚠ 降权`，后续执行时跳过该方法（除非用户显式指定）
- 连续成功 ≥ **5 次**（从降权状态恢复后）→ 恢复为 `✅ 正常`
- 总失败率 > **70%** 且总调用 ≥ **10 次** → 标记为 `🚫 禁用`

### 集成点

- **路由层检查**：gateway 在执行前读取注册表，跳过降权/禁用的方法
- **收束更新**：flow-behavior-auto-receipt 写入快照时自动更新注册表
- **跨设备**：待定——不同设备环境差异可能导致注册表误导

## 组件 B：能力模式识别与固化

### B1 被动模式 — 用户显式固化

用户要求「封装这个能力」：提取步骤序列 → 生成 flow-*.mdc → 登记别名 → 回执确认。

### B2 主动模式 — 自动模式发现（二期）

- **检测窗口**：每 N 轮（建议 N=10）或每次 /收束 时扫描
- **检测对象**：Round-Behavior-Log 的活跃快照表
- **触发阈值**：同一任务类型的工具通道序列出现 ≥3 次 → 提示；≥5 次 → 建议固化
- **固化产物**：轻量追加子节 / 中量新建 flow / 重量升级 mod 或独立 Skill

## 与现有体系的关系

```
gateway-command-router（路由时读取可靠性注册表）
       │
  ┌────┴────┐
  │ Method-Reliability-Registry (新增)
  │ Capability-Registry (新增)
  └────┬────┘
       │
  Round-Behavior-Log（扩展 method_outcome 维度）
       │
  flow-behavior-auto-receipt（收束时更新注册表）
```

- 继承自 T001「行为偏好系统改造」的 Round-Behavior-Log 和 Active-Task-Tracker
- 与「三层架构：编排调度—能力—执行」概念对齐：注册表属于能力层元数据，模式固化属于执行层管道化

## 待决策项

> ✅ 全部已确定（2026-05-10）：
>
> 1. **阈值**：3 次降权 / 5 次恢复 / 70%+10 次禁用，硬编码
> 2. **B2 触发**：每次收束时 AskQuestion `[扫描模式] [跳过]`
> 3. **固化物形式**：≤3 步 → 别名组合，4-7 → flow-*.mdc，8+ → Skill
> 4. **跨设备**：随 Git 同步 + 设备标识列（hostname），路由时只读取当前设备记录
