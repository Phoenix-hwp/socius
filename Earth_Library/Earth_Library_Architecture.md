# Earth Library 架构设计

> 定位：为本工作区所有项目提供**知识参考补充层**，作为网络信息的辅助与沉淀。

---

## 1. 架构定位

### 1.1 角色定义

| 层级 | 来源 | 优先级 | 用途 |
|:---|:---|:---:|:---|
| **主层** | 模型知识 + 网络实时信息 | 1 | 默认回答来源，实时、通用 |
| **补充层** | **Earth Library** | 2 | **个人/项目专属知识沉淀**，上下文连贯 |
| **档案层** | 项目专属文档（如 469_Sports_ProjectContext.md）| 3 | 项目长期背景与决策记录 |

### 1.2 核心职责

1. **知识沉淀**：将对话中的新增知识、网络搜索精华、读书笔记等入库
2. **关联网络**：自动建立知识间的关联（关键词/标签/引用）
3. **检索服务**：为其他项目提供知识检索与推荐
4. **质量维护**：巡检、纠错、优化，保持知识库健康

## 3. 与其他项目的集成

### 3.1 集成模式

| 模式 | 触发时机 | 说明 |
|:---|:---|:---|
| **被动查询** | 用户提问时 | Agent 主动搜索 Library 补充回答 |
| **主动推送** | 入库时 | 根据标签/关键词推送到相关项目 |
| **项目订阅** | 项目初始化时 | 项目声明关注特定标签/领域 |

### 3.2 与 469_Sports 项目的关系示例

```
用户提问："盈钻分佣规则是什么？"

回答优先级：
1. 【主层】模型知识：通用分佣概念解释
2. 【主层】网络搜索：行业常见做法
3. 【补充层】Earth Library：
   - 命中「盈钻」「分佣」关键词的知识卡片
   - 关联到 MEMO-003 分佣基数口径（已确认）
   - 提供「按实付金额计提；盈钻补偿不分佣」等沉淀知识
4. 【档案层】469_Sports_ProjectContext.md：
   - 项目整体背景与业务目标
```

### 3.3 接口设计

#### 检索接口

```python
# 关键词搜索
python Earth_Library/scripts/search_library.py \
  --q "盈钻 分佣" \
  --max_results 5 \
  --include_related true

# 标签筛选
python Earth_Library/scripts/search_library.py \
  --tag "产品设计" \
  --tag "高置信度" \
  --confidence_threshold "中"
```

#### 关联推荐接口

```python
# 给定知识卡片，返回关联卡片
python Earth_Library/scripts/get_related.py \
  --card_path "Knowledge_Cards/20260508-205831_SWOT分析法.md" \
  --relation_types "标签相交,关键词相交"
```

## 5. 知识质量与生命周期

### 5.1 置信度体系

| 置信度 | 定义 | 使用建议 |
|:---:|:---|:---|
| **高** | 已验证、多来源交叉确认 | 可直接引用 |
| **中** | 单来源但可信、或待进一步验证 | 引用时注明来源 |
| **低** | 初步信息、猜测、待确认 | 仅作为参考线索 |

## 6. 使用场景

### 6.1 场景1：对话中沉淀知识

```
用户："存入图书馆：盈钻补偿不参与分佣，退款冲销按原分佣金额全额冲回"

Agent：
1. 解析内容 → 提取字段
2. 自动标签 → ["盈钻", "分佣", "退款", "高置信度"]
3. 生成卡片 → Knowledge_Cards/20260503-xxx.md
4. 更新索引 → Library_Index.md
5. 计算关联 → Relations_Index.md（关联到 MEMO-003）
6. 回执确认 → "已入库，关联到 3 条现有知识"
```

### 6.2 场景2：回答时引用知识

```
用户："盈钻分佣规则是什么？"

Agent（图书馆已启用）：
1. 主层回答：通用分佣概念
2. 搜索 Library：命中「盈钻」「分佣」相关卡片
3. 补充引用："根据此前沉淀的知识（Library/20260503-xxx），盈钻补偿不参与分佣..."
4. 给出结论：综合模型知识与沉淀知识
```

### 6.3 场景3：项目间知识复用

```
用户："新项目需要设计分佣系统，有没有参考？"

Agent：
1. 搜索 Library：标签="分佣" OR 关键词="佣金"
2. 返回相关卡片：469_Sports 的盈钻分佣规则、其他项目类似设计
3. 推荐知识："发现 2 个相关案例可供参考..."
```

## 7. 两种入库模式对比

Earth Library 支持两种不同的 Notion 入库策略，根据知识结构目标选择：

### 7.1 归纳入库（汇总模式）

**适用场景**：快速归档整本书/课程，保留完整概览。

**执行入口**：`notion_drill_earth_library.py`

**产出**：
- 一张大卡片包含整棵树的内容
- 子页面内容以章节列表形式汇总在 Details 中
- 适合「快速备份、全文可检索」

**空间占用**：较大（包含所有子页面文本）

**网状结构**：整本书作为单一节点

---

### 7.2 原子入库（拆分模式）

**适用场景**：构建概念级知识网络，一章/一主题 = 一个独立节点。

**执行入口**：`notion_atomic_ingest.py`

**产出**：
- 每个 Notion 页面生成一张独立卡片
- 卡片结构：
  ```
  # Summary（一句话概括）
  # Details
    ## 核心要点（3-5点，人工后续补充）
    ## 关键摘录（少量原文+位置）
    ## 溯源（Notion链接）
    ## 本地补充（人工自由编辑区）
  ```
- 不存储全文，通过链接溯源

**空间占用**：轻量（单卡 < 2KB）

**网状结构**：
- 每张卡片是独立节点
- 支持章节间建立 `precedes`/`follows`/`prerequisite` 等语义边
- 与既有关联（关键词/标签相交）叠加

---

### 7.3 选择建议

| 你的目标 | 推荐模式 |
|:---|:---|
| 想快速归档，全文本地可查 | 归纳入库 |
| 想构建可导航的知识网络 | **原子入库** |
| 担心占用太多空间 | **原子入库** |
| 后续需要逐章精读、写笔记 | **原子入库** |
| 只是临时备份，不关心关联 | 归纳入库 |
