### MEMO-003：数据治理宪法设计方案

**日期**：2026-05-13

**背景**：基于全库数据格式审计（含 46 JSON / 6 JSONL / 55 MDC / 127 MD），发现当前存在 JSON 嵌长文本、双轨存储、格式异构、字段命名不一致等问题。趁数据量尚小时建立底层治理规范，避免日后大规模重构。

**宪法三原则**：

| 原则 | 规则 | 审计中发现的违规项 |
|:---|:---|:---|
| **原则 1：JSON 不存人类阅读的长文本** | JSON 仅存元数据 + 枚举值 + 指针；正文落 `.md` | `cards.jsonl` 的 `body_md` 嵌几千字 Markdown；`library_index.json` 的 `keywords` 含长句 |
| **原则 2：一份数据只有一个权威源** | 索引只存指针不存正文；可运行时计算的不持久化 | `cards.jsonl` ↔ `library_index.json` 双轨；`relations.jsonl` 83 万字符的 N×M 自动关联 |
| **原则 3：格式一致性是第一公民** | 同一文件内所有行共享字段集合；可选用 null；写入前验证 schema | `Behavior-Fit-Log.jsonl` 双格式混合；`Round-Behavior-Log.jsonl` hot/warm 字段异构 + 格式错误行；`.md` frontmatter 命名不一致 |

**格式分配表**：

| 格式 | 用途 | 禁止 |
|:---|:---|:---|
| **JSON** | 配置、注册表、索引、任务追踪 — 字段可枚举、可检索 | 不存长文本（>50 字）、不存 Markdown |
| **JSONL** | 行为日志、关联数据 — 行式追加，固定 schema | 不混用多个 schema、不存可运行时计算的数据 |
| **.md / .mdc** | 知识正文、规则、备忘 — 需要 AI 语义理解 | 不被 JSON 包裹 |
| **.txt** | 白名单、一次性说明 | 不替代 .md 的功能 |

**落地计划**：

| 阶段 | 动作 | 关联待办 |
|:---|:---|:---|
| **宪法落地**（本轮） | 创建 `data-governance-standards.mdc`（alwaysApply: true） | ✅ 已完成 |
| **紧急修复**（本轮） | 修 `Round-Behavior-Log.jsonl` 格式错误 + 收束加 §5 兜底 | ✅ 已完成 |
| **格式统一**（近期，5-16） | Behavior-Fit-Log.jsonl legacy→compact 格式统一 + .md frontmatter 命名风格统一 | P005, P006 |
| **核心迁移**（中期，5-25） | cards.jsonl → `Earth_Library/cards/<id>.md`；relations.jsonl 瘦身；library_index.json keywords 清理 | P003 |

> **2026-05-13 调整说明**：原 P003 打包了所有数据治理迁移项，经回顾发现不合理的两点——(1) 格式统一（P005/P006）风险低且每天都在累积 tech debt，应提前执行而非等到 12 天后；(2) 核心数据迁移（P003 缩窄）依赖宪法规则稳定后再动手，5-25 排期合理。故拆分 P003 → P003(核心迁移) + P005(Behavior-Fit-Log) + P006(frontmatter)。

**关联待办**：[P003, P005, P006]

### MEMO-004：三个架构扩展方向讨论

**日期**：2026-05-13

**背景**：对方提出了三个系统进化方向的建议，本轮逐一讨论并形成初步判断，留待明天继续深化。

---

#### 方向 1：Skills Library → 能力虚拟化（Capability Virtualization）

**现状**：Skills Library 已有准入评分、安全闸门、隔离试运行、执行前后审查、生命周期管理。但本质上仍是"技能 = 外部脚本"的插件系统。

**建议核心**：
- 定义统一的 `Capability Interface`：输入 schema、输出 schema、副作用声明、依赖声明
- 内核只与"能力接口"交互，不与具体脚本绑定
- 效果：多实现（同一能力的多个技能）、能力路由（根据表现评分自动选择最优实现）

**当前差距**：
- `skill-registry.json` 有 `skills` 列表但无标准化的 `input_schema / output_schema / side_effects / dependencies` 字段
- `flow-skill-execute.mdc` 按注册表直接调用，无路由层
- 无能力抽象层（同一能力的多种实现无法共存）

**初步判断**：方向正确，但实现复杂度较高。建议分阶段：先定义接口规范 → 改造现有注册表 → 新增路由层。明天讨论排期。

---

#### 方向 2：Earth Library → 语义关系图（Semantic Graph）

**现状**：Earth Library 已有原子入库、归纳入库、标签体系、置信度体系、巡检机制。`relations.jsonl` 已存储 `precedes / follows / prerequisite / supports / contradicts` 等关系，但当前 83 万字符体量过重（含大量 N×M 自动生成的低质量边）。

**建议核心**：
- 每次入库时自动生成关系边
- 每次检索时根据关系图扩展上下文
- 每次巡检时检查关系冲突

**当前差距**：
- 关系生成已有，但瘦身策略未执行（MEMO-003 的核心迁移包含此项）
- 检索时的关系图扩展上下文功能尚未实现（`flow-library-search.mdc` 目前仅做关键词匹配）
- 关系冲突检测未纳入巡检流程

**初步判断**：这是三个方向中最接近当前能力的。核心迁移（P003，5-25）已覆盖关系瘦身，检索扩展和冲突检测可作为下一阶段增量。项目自身资料的关系图逻辑可以复用同一套基础设施。明天讨论优先级。

---

#### 方向 3：决策偏好模型（Decision Preference Model）

**现状**：行为偏好系统已有 14 维度行为追踪、双层快照（hot/warm）、三层认知分析。但当前仅用于"观察与建议"，未授权 Agent 自主决策。

**建议核心**：
- 从行为偏好数据中构建决策偏好模型
- 制定决策分级策略：轻量级决策交给 Agent 直接执行，中等以上由 Agent 提供建议、用户决策
- 或按任务类型划分授权边界

**当前差距**：
- 行为数据已有，但未提炼为"决策授权模型"
- 无"红线"机制（哪些事 Agent 绝不可自行决定）
- 无"停止"机制（Agent 在何种条件下必须暂停并询问）
- 当前 `method-reliability-registry.json` 有 ⚠ 降权/🚫 禁用标记，可作为起点

**初步判断**：这是三个方向中**最敏感、最高风险**的，但也是最有长期价值的。必须谨慎设计边界，建议先定义"红线清单"和"分级矩阵"，在有充分数据支撑前保持人工兜底。明天讨论切入方式。

---

#### 讨论总结

| 方向 | 当前成熟度 | 与现有待办关联 | 建议优先级 |
|:---|:---|:---|:---|
| 能力虚拟化 | 低（接口层缺失） | 全新工作 | 长期（先规范后落地） |
| 语义关系图 | 中（基础已有，缺检索扩展） | P003（核心迁移） | 中短期（用 P003 打底） |
| 决策偏好模型 | 中（数据已有，缺授权框架） | `method-reliability-registry` | 中期（需红线设计先行） |

---

#### 方向 4：任务排期能力（Pending-Plan-Tracker 升级）

**现状**：`Pending-Plan-Tracker.json` 已有待办追踪（到期提醒、关键词匹配、自动完成检测），但未被封装为 skill，且缺乏排期维度的字段。

**核心问题**：当前"待办追踪"≠"任务排期"。缺失：
- 任务间依赖声明（P003 阻塞 P007 时需手动判断先后）
- 工作量估算（无 `estimated_hours` 字段，无法判断一周是否过载）
- 阻塞检测（被依赖项延期时无自动提示）
- 视图（无本周视图 / 逾期汇总 / 被阻塞项清单）

**建议边界**：
- **值得做（低风险、高价值）**：`depends_on` 依赖字段 + `estimated_hours` 工作量字段 + 封装为 skill（`mod-task-tracker-framework.mdc`）+ 本周视图工作流
- **守住不做（中高风险，依赖决策偏好模型）**：自动级联重排、甘特图可视化、多项目资源平衡
- 积累的工作量数据可后续为决策偏好模型提供"排期可靠性"维度训练

**初步判断**：与方向 1-3 不同，方向 4 更接近"对现有系统的增量升级"而非全新构造。建议一步到位封装为 skill，在此基础上渐进加字段。明天讨论是否与 P007 合并或拆分独立待办。

### MEMO-004 续：2026-05-14 深化讨论 — 权限决策框架 + 任务分解方案

> 本轮聚焦方向 3（决策偏好模型），产出了完整的权限评估维度、等级推导与落地路径。同时讨论了任务分解与排期的数据架构预留。

#### A. 决策偏好模型 → 权限决策框架（Decision Framework）

**核心理念**：从行为偏好观察升级为任务级决策授权。Agent 按任务维度评估风险/创意/模糊度/依赖，自动推导 L0-L3 权限等级。

##### A1. 决策维度（7 维，其中 4 核心 + 3 现有注入）

| 维度组 | 维度 | 值域 | 来源 | 作用 |
|---|---|---|---|---|
| **R — 风险** | 操作影响面 | R0 只读 / R1 可逆写入 / R2 跨模块 / R3 不可逆 | **新增** | 决定基础权限等级 |
| **C — 创意** | 发挥自由度 | C0 模板 / C1 框架内 / C2 开放式 | **新增** | C2 即使低风险也建议告知 |
| **S — 模糊度** | 目标清晰度 | S0 极清晰 / S1 方向有细节缺 / S2 目标模糊 | **新增** | S2 → 强制 L3（仅分析） |
| **D — 依赖** | 影响范围 | D0 单模块 / D1 多模块 / D2 跨系统 | **新增** | D2 → 至少 L2 |
| **M — 方法可靠性** | 该方法历史表现 | `连续成功/失败次数` + `state` | **现有** `method-reliability-registry.json` | FSM 升级降级权重 |
| **P — 路径敏感度** | 目标路径类型 | `normal` / `.cursor/rules/` / `10-Topics/` / `Earth_Library/scripts/` | **现有** `flow-high-risk-safety.mdc` | 规则层/数据核心 → 强制 L3 |
| **E — 外部依赖** | 是否涉及外部服务 | `none` / `MCP` / `tunnel` / `external_service` | **现有** `external-dependency-boundary.mdc` | 外部依赖 → 至少 L1 |
| **X — 拆解复杂度** | 是否需拆分子任务 | X0 单任务 / X1 可拆未拆 / X2 需拆解 | **新增预留** | 未来自拆解能力的基础 |

##### A2. 权限等级推导（L0-L3）

```
L0（完全托管）：R0 ∧ C≤1 ∧ S0 ∧ D0 ∧ M=正常 ∧ P=normal ∧ E=none
L1（告知执行）：满足任一：R1 ∨ C1 ∨ S1 ∨ D1 ∨ E≠none，且不触发 L2/L3
L2（确认执行）：满足任一：R2 ∨ C2 ∨ D2 ∨ M=降权 ∨ P=.cursor/rules/，且不触发 L3
L3（仅分析）：满足任一：R3 ∨ S2 ∨ P=数据治理核心 ∨ M=禁用
```

##### A3. 任务级 vs 命令级对比

| | 命令级（现状） | 任务级（方案） |
|---|---|---|
| 上下文 | 每条命令独立判断 | 一次评估整个任务 |
| 语义 | 看不出改 typo vs 改架构 | 四维度区分意图本质 |
| 确认 | 高低风险同等弹窗 → 麻木 | 任务开始时统一策略 |
| 进化 | 无历史决策记忆 | FSM 同类任务连续成功 → 升级放权 |

**核心权衡**：任务级决定默认行为，命令级（`flow-high-risk-safety.mdc` 红色警戒）做兜底阻断，双层嵌套不互替。

##### A4. 决策日志 schema

```json
{"date":"ISO","task_type":"<摘要>","dimensions":{"R":0-3,"C":0-2,"S":0-2,"D":0-2},"derived_level":"L0-L3","override":null,"user_intervention":false,"outcome":"success|partial|failure"}
```

#### B. 任务分解与排期架构

##### B1. 正确的层级链路

```
目标/父任务（Active-Task-Tracker）
  ├─ 子任务 A → 多个待办（逐待办执行）
  ├─ 子任务 B → 依赖 A 完成 → 排期自动推进
  └─ 子任务 C → 并行或串行
```

##### B2. 数据 schema 预留

**Active-Task-Tracker 加字段**：
- `parent`：父任务 ID（null = 根任务）
- `children`：子任务 ID 列表
- `next_child`：当前待推进的子任务（串行用）

**Pending-Plan-Tracker 加字段**：
- `source_task`：关联的子任务 ID

**新建 `Decomposition-Log.jsonl`**：拆解决策日志（当前不写入，为自拆解能力预留数据源）

##### B3. 自拆解能力的路径

```

  ↓ 连续 N 次用户接受且未大幅修改
X1 → X2 自动拆解
```

### MEMO-004 续：2026-05-14 第三轮 — 任务 I/O：内容结构决定输出一致性约束

> 本轮讨论了任务输入输出的隐式约束问题 —— 不是文件格式决定约束，而是「输入内容是否有非平凡结构」决定输出是否需要保持同构。

#### A. 核心观点

**约束的来源不是文件扩展名，是内容的排版复杂度。**

同是 `.md` 文件：
- 一段纯文本日志 → 怎么改都行，无结构 = 无约束（C2）
- 有标题层级 + 表格 + 图片的 README → 改内容但排版本必保留（C0）

PDF 之所以几乎总是隐含"排版不可变"，不是因为它是 PDF，而是因为 PDF 几乎总是有排版过的结构。但反过来，一个有精心排版内容的 Word/HTML，同样有排版约束。

#### B. 对决策框架的影响

在 S 维度（目标清晰度 / 语义模糊度）评估时，增加一个子判断：

> **Agent 扫描输入内容：是否存在非平凡结构？**
> - 标题层级 ≥ 2 级
> - 表格
> - 图片/插图
> - 列表结构
> - 分页/分栏/特定区域
>
> → 有结构 → 输出须保持同构（版式一致），创意自由度压为 C0
> → 无结构 → 输出自由，C 维度按任务意图正常评估

此判断不新增维度，作为 S 维度和 C 维度的辅助输入。

#### C. 更一般的 I/O 约束框架（待深化）

| 输入特征 | 隐含约束 | 对决策的影响 |
|---|---|---|
| 有排版结构的文档 | 输出保持同构 | C → C0，D 维度关注输出工具链的排版保真度 |
| 无结构的纯文本 | 无约束 | C 按任务意图自由评估 |
| 图片/设计稿（纠错类） | 构图不变 | C → C0 |
| 图片/设计稿（风格迁移） | 允许换构图 | C → C2 |

此框架尚不完整，其余任务类型（代码、配置、数据库等）待后续补充。
